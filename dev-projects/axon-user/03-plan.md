## 0. Note on plan generation (why no `code-dev plan` here)

This plan was **not** generated by the `code-dev plan` program. The persona
runs were driven by sub-agents in read-only mode and synthesized into findings
manually. The richer artifacts a real `code-dev plan` invocation would have
produced were therefore missing in the first draft:

| `code-dev plan` artifact                                     | first draft | now |
|--------------------------------------------------------------|-------------|-----|
| Wave-grouped tables with per-PR detail-file links            | ✗           | ✓   |
| Per-PR `.md` files (one per PR, 9-section schema)            | ✗           | ✓   |
| `DAG.md` with Mermaid graph + critical path                  | ✗           | ✓   |
| `DAG.json` with edges + topo                                 | ✗           | ✓   |
| Governance trace (rules read + filtered + flagged)           | ✗           | ✓   |
| MUST/NICE split per wave                                     | ✗           | ✓   |
| Risk register                                                | ✗           | ✓   |
| Acceptance criterion per PR + global `_check-all.sh` gate    | ✗           | ✓   |

The root cause is a **process gap**: I treated "write a plan" as "write a
table" rather than as "produce the full `code-dev plan` deliverable set". The
errata below corrects the deliverable, not the underlying findings.

Separately, this also exposes a **product gap** in AXON itself:
`code-dev plan` requires an interactive turn with a real user agent and lives
in the program executor's runtime — there is no offline "dry-render the plan
artifacts from a fixture" mode. That observation is logged as out-of-scope
(see [findings/out-of-scope.md](findings/out-of-scope.md)) because adding a
dry-render mode is a feature, not an improvement.

## 9. Why each PR matters (executive context)

Per-PR rationale lives in each detail file's `## Why` section. The
high-level theme is:

- **U-1, U-2, U-3, U-5**: dispatch-layer integrity. PR-26/27/28 changed the
  user-visible surface but didn't propagate the rename through file bodies
  and tool contracts. Until U-A/U-B land, the W4 rename benefit is mostly theatrical.
- **U-4**: removes a misleading promise. Subtractive fix.
- **U-6, U-7**: small wins that compound — remove duplicate work
  (branch-sync, blanket-budget) that costs tokens and confuses users.
- **U-8, U-9**: discoverability. Each is small individually but collectively
  cost a first-day user 30-50% of their patience budget.

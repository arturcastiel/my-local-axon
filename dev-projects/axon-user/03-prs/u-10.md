# U-10 — `code-dev plan --mode=strategic` writes `02-roadmap.md`

**Wave**: U.E (planning hierarchy) · **Depends-on**: U-1
**Owner**: AGENT writes; HUMAN runs a dry plan against a fixture project
**Status**: ready

## Why

Today `--mode=strategic` only prints a stdout banner. There is no persisted
roadmap artifact. A roadmap is the natural home for cross-plan / cross-release
vision — multiple plans (e.g. 3.6.1 errata + 3.7.0 feature work) should hang
under one roadmap. Without it, the "long view" lives only in CHANGELOG.

## Evidence

- [03-plan-v3.md §1.2](../03-plan-v3.md) — current `--mode=strategic` is
  stdout-only.
- `workspace/programs/code-dev-plan.md` lines ~40-55 (MODE DISPATCH block).

## Design notes

- Add `IF plan-mode ≡ "strategic"` branch that writes `02-roadmap.md` using
  a new template `workspace/templates/v4-roadmap.md`.
- Roadmap content sections (template-enforced):
  1. **Vision** — 2-4 sentences. What does this product / project become?
  2. **Releases** — table of upcoming releases with theme + target date (optional).
  3. **Plans hosted** — list of `02-plan.md` / `02-plan-v*.md` documents under
     this roadmap, with one-line summary each.
  4. **Out-of-roadmap** — explicitly named items that are *not* on the trajectory.
  5. **Iteration log** — append-only delta history.
- Strategic mode still produces ≤ 800 tokens to stdout (per existing
  contract); the file output is the rich version.

## Pitfalls

- **R (existing strategic users)**: anyone relying on stdout-only strategic
  output gets an extra file written to disk. Acceptable — non-breaking.
- **F (template drift)**: roadmap template lives next to v4-meta.md and
  v4-schema.md; cross-referenced in U-15.

## Interface sketch

```bash
$ code-dev plan --mode=strategic
✓ wrote my-axon/dev-projects/axon-user/02-roadmap.md  (template: v4-roadmap)
✓ stdout summary follows:

# Strategic plan — axon-user
## Vision
...
```

## Spec

### Files

- `workspace/programs/code-dev-plan.md` — extend strategic dispatch branch
  to WRITE `02-roadmap.md`.
- `workspace/templates/v4-roadmap.md` — **NEW** template (this is allowed:
  templates, not programs).

### Diff sketch (code-dev-plan.md)

```diff
 IF plan-mode ≡ "strategic" →
   → "# Strategic plan — {W:code-dev-project}"
   → "Wave-level overview, ≤800 tokens, no per-PR detail."
   → "Mode: strategic · budget: {budget}"
+  tpl     ← READ("{W:ws-os}/templates/v4-roadmap.md")
+  rendered ← RENDER(tpl, {project, vision, releases, plans, ts})
+  WRITE("{project-dir}/02-roadmap.md", rendered)
+  → ""
+  → "✓ wrote {project-dir}/02-roadmap.md"
```

### Acceptance

```bash
# fixture project with no 02-roadmap.md
code-dev plan --mode=strategic
test -f my-axon/dev-projects/<slug>/02-roadmap.md
head -3 my-axon/dev-projects/<slug>/02-roadmap.md | grep -q "Roadmap"
```

### Rollback

`git revert`. Template file gets removed; roadmap files in user projects are
left in place (they are user data).

### Owner

AGENT writes diff + template. HUMAN runs the fixture acceptance.

## Codebase grounding

- `code-dev-plan.md` already uses `READ` / `WRITE` for `02-plan.md` and
  `02-prs.md`. The new write follows the same pattern.
- v4 template directory `workspace/templates/` already hosts `v4-meta.md`
  and `v4-schema.md`; v4-roadmap.md fits the family.

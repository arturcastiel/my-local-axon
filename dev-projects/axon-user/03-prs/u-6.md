# U-6 — `pr-ready` drop redundant Gate A; rewire to safety-preflight

**Wave**: U.C · **Severity addressed**: S2 (F-003, F-014) · **Depends-on**: U-1
**Owner**: AGENT · **Status**: ready

## Why

`code-dev pr-ready` has 3 top-level gates (A: branch sync, B: working tree
clean, C: preflight passed). Gate C calls `code-dev-preflight`, which runs
its own Gate 0 = branch sync. So branch sync runs **twice** on every
`pr-ready`. Additionally, Gate C calls the deprecation-stub `code-dev-preflight`
which logs a WARN every time, even though the canonical name is now
`code-dev-safety-preflight`.

Two-fold fix: drop Gate A (let preflight handle it), and rewire Gate C
directly to `code-dev-safety-preflight`.

## Evidence

- [findings/F-003](../findings/F-003-preflight-mislabeled.md)
- [findings/F-014](../findings/F-014-pr-ready-redundant-gate.md)

- `workspace/programs/code-dev-safety-preflight.md#L30` (Gate 0 — branch sync).

## Design notes

- Removing Gate A is safe because preflight Gate 0 fires first inside Gate C.
- Switching the EXEC target to `code-dev-safety-preflight` silences the
  alias-stub WARN log; the stub `code-dev-preflight` remains in place for
  any other callers (one release of deprecation).

## Pitfalls

- **D1 (gate order skew)**: if preflight is ever bypassed via `--skip-preflight`,
  pr-ready will not run branch-sync at all. Acceptable: `--skip-preflight`
  already means "user accepts the risk".

## Interface sketch

```bash
$ code-dev pr-ready 17
✓ Gate B  working-tree clean
✓ Gate C  preflight (10 gates) — passed
# Gate A removed — preflight Gate 0 is the single source for branch sync.
```

- `code-dev pr-ready N` log shows exactly one "branch-sync" event.
- No `LOG(WARN, "alias-deprecated:...")` for `code-dev-preflight` during a
  normal `pr-ready` run.
- `tests/test_pr_ergonomics.py` still green.

  has always delegated detailed gates to preflight. Gate A was a defensive
  belt-and-suspenders that became redundant once preflight Gate 0 stabilized.

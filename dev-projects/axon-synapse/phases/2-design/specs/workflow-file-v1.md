# Workflow File Schema (v1)

> glossary: SYNAPSE-GLOSSARY v1
> resolves: Q16.3, D-016, D-017, D-029, D-030
> validated against: WF-01 (code-dev canonical), WF-08 (python-code-dev),
> WF-12 (adaptive free-text)

## Purpose

Define the file format that represents a workflow — the DAG of synapses
with execution-mode, goal, triggers, acceptance, and suggestion controls.

## Where workflow files live

- **Cross-domain workflows:** `workspace/workflows/<name>.yml`
- **Domain-bound workflows:** `workspace/domains/{d}/workflows/<name>.yml`
- **Compiled workflows (Phase 3+):** `workspace/workflows/compiled/<name>.cmp.yml`
  (generated by `workflow-compile`; auto-emitted from adaptive runs that
  the user accepts).

YAML is the canonical machine format. An MD renderer (`workflow-render`)
auto-emits a paired `<name>.md` for human reading.

## Schema (v1) — full reference

```yaml
# Required
name:             python-code-dev
version:          1
domain:           code-dev                    # closed list per glossary
execution-mode:   fixed                       # fixed | adaptive | hybrid

# Goal (per goal-schema-v1)
default-goal:
  statement:      "Ship a Python code change end-to-end."
  measurement:
    - "phase.has(03-prs/PR-*.md) AND all_prs_implemented()"
    - "shadow.coverage(phase) == 100"
    - "tests.pass()"
  acceptance-criterion: "audit.open-findings == 0 AND shadow.coverage == 100"
  rejection-criterion:  "tests.fail() OR audit.critical-issues > 0"

# Discovery / dispatch
triggers:
  - kind: keyword
    when: "user said one of [python workflow, py code-dev, pip change]"
  - kind: state
    when: "project._meta.codebase contains *.py files"
  - kind: explicit
    when: "workflow run python-code-dev"

# Suggestion controls
allow-suggestions: true
allow-deviation:   true
suggestion-channel: [footer, panel]            # subset of [footer, panel, popup]

# Synapse graph — declarative; orchestrator walks it per execution-mode
synapses:
  # NOTE: `python-lint` referenced below is NOT registered today; this example
  # shows the conversational author flow registering it on the fly. Replace
  # with `code-dev-review-self` for an as-is-shippable workflow.

  - id:   s1
    name: code-dev-study
    mode: subsystem
    args:
      target: "**/*.py"
    on-complete:
      - next: s2

  - id:   s2
    name: code-dev-suggest-tests
    on-complete:
      - if: "count(suggested_tests) > 0"
        next: s3
      - if: "count(suggested_tests) == 0"
        next: s5                              # skip to self-review

  - id:   s3
    name: run-tests
    role: gate                                # blocks chain on failure
    on-complete:
      - if: "tests.pass()"
        next: s4
      - if: "tests.fail()"
        next: s3-fix                          # branch — adaptive sub-segment
        mode-override: adaptive

  - id:   s3-fix
    mode-override: adaptive
    description: "Triage and fix failing tests; orchestrator picks next synapse"
    on-complete:
      - if: "tests.pass()"
        next: s4

  - id:   s4
    name: code-dev-review-tests
    on-complete:
      - next: s5

  - id:   s5
    name: code-dev-self-review
    on-complete:
      - next: s6

  - id:   s6
    name: code-dev-explain
    args:
      target: commit-message
    on-complete:
      - next: s7

  - id:   s7
    name: code-dev-pr-create
    on-complete:
      - next: s8

  - id:   s8
    name: code-dev-pr-review
    on-complete:
      - if: "review.passes()"
        next: s9
      - if: "review.has-objections()"
        next: s8                              # re-run review until passes

  - id:   s9
    name: code-dev-safety-audit
    on-complete:
      - if: "audit.open-findings == 0"
        next: s10
      - if: "audit.open-findings > 0"
        next: s9-fix
        mode-override: adaptive

  - id:   s9-fix
    mode-override: adaptive
    description: "Address audit findings"

  - id:   s10
    name: code-dev-knowledge-shadow
    on-complete:
      - next: s11

  - id:   s11
    name: code-dev-finalize                   # depends on F-012 stub closure
    on-complete: []                           # terminal

# DAG entry point
start: s1

# Per-step execution-mode override (for hybrid workflows)
# Default: workflow.execution-mode applied to every step.
# Overrides expressed via per-synapse `mode-override:` field above.

# Metadata
author:           "axon-synapse Phase 2 design"
created:          2026-05-17
tags:             [code-dev, python, fixed]
parent-workflow:  code-dev-canonical          # inherits sub-graph; this is a refinement
```

## Per-mode field semantics

### Fixed mode

- The DAG is walked sequentially per `on-complete.next` edges.
- The orchestrator does NOT consult ranker for next-step selection (path
  is declared).
- Suggestions remain live per `allow-suggestions: true` — surfaced sideband.
- Deviation requires explicit user accept (per `allow-deviation: true`).

### Adaptive mode

- The DAG declares `start` but `synapses:` may be incomplete.
- At each step the orchestrator: observes state, ranks candidates, fires
  top-1 (subject to `L:inference-mode` confirmation).
- `default-goal.acceptance-criterion` is the loop exit predicate.
- Authoring tip: declare `triggers` precisely; leave `synapses` empty
  unless seeding the orchestrator with anchor candidates.

### Hybrid mode

- Per-synapse `mode-override: adaptive | fixed` flips local behavior.
- Useful for: fixed entry steps, adaptive triage segment, fixed exit steps.

## Adaptive sub-segments inside Fixed workflows

The `s3-fix` and `s9-fix` examples show **adaptive sub-segments** inside a
Fixed workflow. The orchestrator:
1. Observes triage state (which tests failed, which findings opened).
2. Ranks candidate synapses (e.g. `code-dev-review-coverage`,
   `code-dev-impact`, `code-dev-test-map`).
3. Fires top-1 with user confirm (per inference-mode).
4. Loops until the segment's `on-complete.if` predicate holds.
5. Returns to fixed path at `next:` target.

## Suggestion controls — details

### `allow-suggestions`

- `true` (default) — sideband suggestions live in chosen `suggestion-channel`.
- `false` — orchestrator suppresses all suggestions during this workflow.

### `allow-deviation`

- `true` (default) — user may accept a deviation suggestion mid-fixed-path.
- `false` — Fixed path is enforced; deviations refused.

### `suggestion-channel`

Closed list:
- `footer` — single-line in output-layer footer.
- `panel` — multi-line panel above prompt.
- `popup` — modal QUERY before next fire (high friction; use sparingly).

## Validation (Phase 3 deliverable: `workflow-validate`)

A workflow file is valid iff:

1. Every `name` in `synapses[]` exists in synapse registry OR is a
   declared sub-id (e.g. `s3-fix`).
2. `start` references a real `id`.
3. DAG is acyclic (cycle-detector matches `plan_dag.py` Kahn algorithm).
4. Every `on-complete.next` references a real `id` (or `[]` for terminals).
5. `domain` is in the closed list.
6. `execution-mode` is in {fixed, adaptive, hybrid}.
7. Predicates in `if:` clauses parse against predicate-language-v1.
8. `default-goal` is non-empty and parseable per goal-schema-v1.

## Authoring paths (per D-016)

### Direct authoring

User writes the YAML by hand.

### Conversational authoring (per D-028)

Entry: `workflow-new --from-description "<text>"`.

Interactive dialog (sketched in conversational-author-v1.md):

```
USER: I want a python workflow that lints, tests, reviews, and writes
      commit msg.
AXON: I'll author this as a Fixed workflow. Domain: code-dev. Goal:
      "Ship a Python code change with lint+test+review+commit-msg."
      OK?
USER: yes
AXON: Proposed first synapse: code-dev-study --mode=subsystem (read the
      changed code). OK?
USER: yes
AXON: Proposed next: a python-lint synapse. I don't see one registered.
      Should I register a stub for you to fill in, or pick an existing
      one (code-dev-review-self runs lint heuristics)?
USER: register stub
AXON: [calls register-tool wizard for python-lint] ...
... continues until DAG complete ...
AXON: Final workflow:
      [renders YAML]
      Save to workspace/domains/code-dev/workflows/python-code-dev.yml?
USER: yes
AXON: Saved. To run: workflow run python-code-dev
```

## Compilation (optional optimization, Phase 3+)

`workflow-compile <name>` produces `workspace/workflows/compiled/<name>.cmp.yml`
with: pre-resolved synapse-contracts inlined, predicate AST cached,
critical-path computed. Subsequent runs use the compiled form.

## v1.1 additions (2026-05-17)

### `execution-mode` values expanded (GAP)
Closed list: `fixed | adaptive | hybrid | exploratory | scheduled`.
See AXON-GLOSSARY v2 for semantics.

### Cross-domain workflows (GAP-01)
`domain:` field accepts a list:
```yaml
domain: [code-dev, library-dev]
```
Resolution: first-listed domain's manifest is primary; subsequent
manifests provide secondary verb-maps. Naming collisions in verb-map
resolved by primary.

### Mid-workflow mode switching (GAP-02)
Each step may declare:
```yaml
- id: s4
  name: code-dev-pr-review
  mode-switch: ["fixed→adaptive", "adaptive→fixed"]   # allowed transitions
```
Defaults to `[]` (no mid-step switching).
A switch requires explicit user-confirm via QUERY OR an `on-complete.if`
clause that names the switch.

### Suggestion budget (GAP-03)
```yaml
suggestion-budget:
  sideband-per-step: 1
  sideband-per-run:  10
  dismiss-decay:     0.5     # after dismiss, weight halves for 30 min
  rate-limit-window: "5min"
```
Suppresses suggestion noise. Per-workflow override.

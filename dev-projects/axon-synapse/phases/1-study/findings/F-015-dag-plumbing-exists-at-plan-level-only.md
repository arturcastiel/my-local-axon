## Why this matters

- Per D-009 (DAG central at every level), the project requires DAG files at
  5 levels with nested sync validation. Today only 1 level has them, and
  even that level is manually maintained.
- The plumbing tool (`plan_dag.py`) is solid — Mermaid + JSON + Kahn
  cycle-check + critical-path. **Not** a build-from-scratch situation.
  It's a **wire-up** situation.
- Backwards compat (D-014/D-025) — existing plan-level DAG files stay
  in `03-prs/DAG.{json,md}` form. New levels add new files.

## Implication for Phase 2 / Phase 3

- **Phase 2.** Spec the 5-level DAG file layout:
  - Project: `<project>/DAG.{json,md}` (phase graph; replaces /
    complements `masterplan.md`).
  - Phase: `<project>/phases/{n}/DAG.{json,md}`.
  - Plan: existing `<project>/{phase}/03-prs/DAG.{json,md}` or
    `<project>/03-prs/DAG.{json,md}`.
  - PR: `<project>/{phase}/03-prs/PR-NNN/DAG.{json,md}` (optional;
    only if PR is subtasked).
  - Study: `<project>/phases/1-study/DAG.{json,md}` (research-Q DAG).

- **Phase 2.** Spec sync-checker semantics. Candidate algorithm:
  - Source of truth: `DAG.json` per level.
  - Renderer: `DAG.md` auto-regenerated from `.json` (one-way: json → md).
  - Hand-edits to `.md` are NOT round-tripped (avoids the round-trip
    complexity flagged in D-006). User edits `.json` directly or via
    the mutator programs.
  - Nested consistency: child-DAG file existence ↔ parent-DAG edge target.

- **Phase 3 PR seeds:**
  - `plan_dag-auto-emit-hook` — `code-dev plan` finalize step calls
    `plan_dag` automatically; writes both files.
  - `dag-sync` tool — verifies nested DAG consistency; reports drift.
  - `dag-mutate-on-{combine,divide,fold,defer,cut,rename}` — every PR
    mutator program calls `dag-sync` to update + verify.
  - `dag-bootstrap-project / -phase / -study / -pr` — generate empty
    DAG files at each level on container creation.

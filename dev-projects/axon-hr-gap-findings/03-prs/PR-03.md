## Summary

The OS STATE panel in `workspace/programs/menu.md` (lines 159-162) displays a
health score with grade bars and labels (Excellent / Good / Fair / Poor) but
gives no indication of HOW the score was computed. The score comes from
lightweight smoke probes, not a full test suite — a reader seeing "Health ●●●●●
100/100 Excellent" could reasonably interpret this as "all tests pass."

Fix: add `(smoke)` qualifier to the "Health" label in all 4 tier renders. Zero
logic change. The ELSE branch (line 163: "Health unknown") is left unchanged —
when no score exists, there is nothing to qualify.

## Changes Required

### workspace/programs/menu.md — lines 159-162 (4 render strings)

**What:** Insert ` (smoke)` after `Health` in each of the 4 tier render strings.
Replace the spacer padding to maintain a clean display format.

**Before (exact, lines 159-162):**
```
  IF hscore >= 90 → → "  Health       ●●●●● {hscore}/100  Excellent"
  IF hscore >= 70 → → "  Health       ●●●●○ {hscore}/100  Good"
  IF hscore >= 50 → → "  Health       ●●●○○ {hscore}/100  Fair  ⚠ run stats"
  IF hscore <  50 → → "  Health       ●●○○○ {hscore}/100  Poor  ! run health-check"
```

**After:**
```
  IF hscore >= 90 → → "  Health (smoke) ●●●●● {hscore}/100  Excellent"
  IF hscore >= 70 → → "  Health (smoke) ●●●●○ {hscore}/100  Good"
  IF hscore >= 50 → → "  Health (smoke) ●●●○○ {hscore}/100  Fair  ⚠ run stats"
  IF hscore <  50 → → "  Health (smoke) ●●○○○ {hscore}/100  Poor  ! run health-check"
```

**Unchanged (line 163):**
```
ELSE → → "  Health       unknown  (run: health-check)"
```

**Why:** `(smoke)` signals that the score is from lightweight probes, not a full
test run. Honest display prevents misreading a high score as test coverage
confirmation. The unknown branch is already self-describing ("unknown") and
has no numeric value to qualify.

**Implementation note:** Use one Edit call with `old_string` = the exact 4-line
block (Before section above) and `new_string` = the 4-line block with `(smoke)`
added. Single atomic change — safest approach.

Do NOT use `replace_all=True` here. That flag means "replace every occurrence of
this exact string in the file" — appropriate for a single repeated token (like
PR-02's base-path replacement). It does not apply when each of the 4 lines
differs; `replace_all` on a multi-line old_string that appears exactly once is
equivalent to a plain Edit but adds risk of unintended matches.

## Risks & Gotchas

- ⚠ **Spacing shift**: Original has 7 spaces between "Health" and "●●●●●"
  (`Health       ●●●●●`). After adding ` (smoke)`, spacing becomes
  `Health (smoke) ●●●●●`. The grade bar shifts 9 chars right. This is intentional
  (label takes precedence over alignment). If column alignment must be preserved,
  reduce trailing spaces after "Health" to compensate — but this is optional polish.
- ⚠ **Single Edit block preferred**: Edit the full 4-line block as one operation.
  Four separate Edit calls each require a unique old_string — less brittle to do it
  as one block replacement.

  contains the same health render block (L140-143). DO NOT update it. The compiler

  are dead artifacts; `run.py` executes `.md` sources directly.

  `workspace/programs/gain.md` (L113-119) renders the same 4-tier health display

  (`→ "  HEALTH  ·  last scored {hscore-date}"`); tier renders L116-119 have no

## Flag-Specific Test Details

### Flag 1: terminal-outputs-required (R_TERMINAL_OUTPUTS)

**What it checks:** When a program reaches a `:done` terminal transition, all
artifacts declared via `# emits:` header must exist on disk.

**Known-bad input for BLOCK test:**
A response containing `:done` with an `active_program` set in the working dir
to a program that declares `# emits:` but whose output file does not exist.

```bash
# Seed the working dir with a fake active program:
echo "test-program" > /tmp/axon-test-ws/memory/working/active-program.md
echo "test-program" > /tmp/axon-test-ws/memory/working/code-dev-project.md
# Test with :done in response (emits file missing = BLOCK):
python3 tools/verify.py output \
  --text "test completed :done" \
  --workspace /tmp/axon-test-ws
```

Note: BLOCK fires only if the active program has a `# emits:` header AND the
declared file is absent. Programs without `# emits:` are unguarded (safe default).

**Risk:** First activation may BLOCK on responses where emits artifacts are
legitimately absent (e.g., in-progress runs, programs with stale emits headers).
If this causes excessive blocking, remove the flag and audit emits headers first.

---

### Flag 2: state-surfaced-required (R_STATE_SURFACED)

**What it checks:** While a non-terminal phase is active
(W:active-phase set, not `:done`/`:failed`/`:aborted`), every response must
include the narrated state block (containing `PHASE` keyword and a forward
pointer `advance`/`next`/`→`).

**Known-bad input for BLOCK test:**
```bash
echo "some-program:start" > /tmp/axon-test-ws/memory/working/active-phase.md
python3 tools/verify.py output \
  --text "Here is my answer to your question." \
  --workspace /tmp/axon-test-ws
# Expected: BLOCK (response lacks PHASE + advance keywords)
```

**Known-good input for PASS test:**
```bash
python3 tools/verify.py output \
  --text "PHASE 01—STUDY status: active → next: run audit" \
  --workspace /tmp/axon-test-ws
# Expected: PASS
```

**Risk:** This flag blocks ALL responses during an active phase that lack the
state block — including tool outputs, follow-up questions, and error messages.
High impact; test thoroughly in an isolated session before enabling in prod.

---

### Flag 3: menu-render-required (R_MENU_RENDERED)

**What it checks:** When `active_program` is `menu` OR the output resembles a
menu (MODES header present), all canonical sections must be present.

Canonical sections: `OS STATE`, `MODES`, `SYNAPSE`, `GOALS`, `NEXT`
(from `workspace/programs/menu.md` Core Rule 12).

**Known-bad input for BLOCK test:**
```bash
echo "menu" > /tmp/axon-test-ws/memory/working/active-program.md
python3 tools/verify.py output \
  --text "## MODES\n- mode A\n- mode B" \
  --workspace /tmp/axon-test-ws
# Expected: BLOCK (partial menu detected — missing OS STATE, SYNAPSE, GOALS, NEXT)
```

**Risk:** Any menu render that omits a section triggers BLOCK. Confirm menu.md
renders all canonical sections correctly (PR-03 should have addressed health
display but not menu structure). Run `menu` program once in isolated workspace
with flag set before enabling in prod.

---

### Flag 4: project-anchor-required (R_PROJECT_ANCHOR)

**What it checks:** When `W:code-dev-project` is set (in working dir), every
response must anchor the active project in its output text.

**Known-bad input for BLOCK test:**
```bash
echo "axon-hr-gap-findings" > /tmp/axon-test-ws/memory/working/code-dev-project.md
python3 tools/verify.py output \
  --text "Here is my answer to your question." \
  --workspace /tmp/axon-test-ws
# Expected: BLOCK (response lacks project anchor)
```

**Risk:** High impact — affects ALL responses while a code-dev project is loaded.
If AXON responds to any message without anchoring the project, it will BLOCK.
Enable last. Test with `code-dev load axon-hr-gap-findings` in an isolated
workspace to confirm the anchor appears in normal responses.

## Risks & Gotchas

- ⚠ **AXON MUST NOT activate any flag**: All flag activation is owner-manual.
  No `echo "true" > ...` from AXON. No scripted activation. Each flag one at a time.
- ⚠ **kv_store.py does NOT set L: flags**: `kv_store.py set --key "L:..." --value true`
  writes to `workspace/memory/kv-store/` (diskcache) — NOT to `longterm/`. The only
  activation path is: `echo "true" > workspace/memory/longterm/<flag>.md`.
- ⚠ **Rollback is fast**: `rm workspace/memory/longterm/<flag>.md` immediately
  disables the flag. No restart needed. The rule reads the file on each `verify output` call.
- ⚠ **terminal-outputs-required fires only at :done**: This is the safest flag to
  enable first. If the active program has no `# emits:` header, the rule is dormant
  even with the flag set.
- ⚠ **state-surfaced-required is high-impact**: It blocks every response during an
  active phase. Test in isolation, not in a live session.
- ⚠ **scripts/enable-enforcement.sh is a no-op (ADR-002)**: Settings files
  (`.claude/settings.json` vs `.claude/settings.json.proposed`) are already identical.
  Running the script makes no change. DO NOT run `--apply`.
- ⚠ **Activation order matters**: Enable flags from lowest to highest impact.
  If `project-anchor-required` is enabled before `terminal-outputs-required`, a
  non-anchored response during code-dev will BLOCK immediately — harder to diagnose.
  Follow the order: terminal-outputs → state-surfaced → menu-render → project-anchor.

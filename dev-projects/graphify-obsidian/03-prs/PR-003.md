## Problem
`tests/test_graphify_bridge.py` pins only the graphify-ABSENT contract (degrade, never
crash). Graphify is now installed on the dev machine (CLI + module), so the live path —
`build` actually producing a graph.json the `affected` query can consume — is exercisable
but untested. A graphify upstream change could silently break the bridge.

## Change
`tests/test_graphify_bridge.py` — new present-path block, every test marked
`pytest.mark.skipif(not gb.available()["available"])` so CI (graphify absent) skips:
- `build` on a tiny fixture repo (2 Python files, one importing the other) returns the
  result envelope; when ok, the graph path exists, parses, and has nodes + links/edges.
- end-to-end: `affected` over the freshly-built graph resolves a symbol from the fixture
  (the bridge's id/label matching against real graphify output, not the synthetic fixture).
- envelope shape stability: keys ok/degraded always present on both paths.

NOT tested (documented): the `semantic` happy path — it needs an AEGIS `web` grant,
network, and token budget. The safety-critical half (refusal without grant, never
EXTRACTED) is already pinned. No assertion on byte-determinism of graphify output —
that's an external tool's property, not the bridge's contract.

## Out of scope
Bridge code changes (none needed unless the present-path tests expose a real defect —
if they do, the fix lands HERE with the test that caught it).

## Gates
R13 tests · lint-paths · freshness · crucible green before merge.

## Acceptance
- Suite green locally WITH graphify installed (present-path runs).
- Suite green with graphify absent (block skips — verified via monkeypatched availability).

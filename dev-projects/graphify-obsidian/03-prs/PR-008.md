## Problem
PR `depends_on` in the plan phase is a pure LLM guess (the goal→files semantic-search
slot has been deprecated since PR-142 — an empty slot). With the per-project graph now
persisted at study (PR-007), real code structure can inform dependency ordering — but
nothing consumes it.

## Change
`tools/graphify_bridge.py` — new subcommand `pr-edges --graph <json> --prs-json <list>`:
- Input: `[{number, files: [...]}]` (per-PR scope files). Maps file→nodes (reuses the
  file_nodes resolution), node→PR; for each graph link from PR-A's nodes into PR-B's
  nodes (A≠B), suggests `A depends_on B` — caller lands after callee. Deterministic
  (sorted), deduped, never self-deps, output tagged `advisory: true` + `confidence:
  INFERRED` (the file→PR mapping is heuristic even though edges are EXTRACTED).

`workspace/programs/code-dev-plan.md` — DAG AUTO-EMIT block:
- IF the project graph exists → `pr-edges` over the PR list's scope files; render the
  hints ("code-derived dependency hints") and MERGE additive-only into the prs-json
  passed to `dag build-from-prs` (existing depends_on never removed; dag.py already
  dedups + drops dangling). dag.py itself untouched — reduce-surface.
- Graph absent → block skipped silently (today's behavior).

Tests: pr-edges over a synthetic two-PR fixture graph (direction: caller depends on
callee), self-dep exclusion, dedup, missing-graph degrade, plan program contains the
merge block (text pin).

## Out of scope
review/test-map consumers (PR-009) · adaptive synapse (PR-010) · dag.py changes.

## Gates

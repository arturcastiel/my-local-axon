# PR-008b — make the advanced _phases.json a VISIBLE consumer (close the dual-SSOT)
Status: merged
Phase: pr
Lane: AXON (autonomous, non-kernel)  ·  Critical-path (after PR-019)

## Problem (gap-find rank 5/6)
PR-008/PR-019 made the ladder ADVANCE `_phases.json` (real state). But NO user-facing program reads it:
`code-dev-state-resume.md:84`, `code-dev-next.md:39`, and `code-dev-state-status.md:41` all read
`_meta.phase` (a single string), never the manifest. So the per-phase advancement (study ✓ → plan ✓ →
pr ◀ → log ⏳ → audit ⏳) the manifest now tracks is REAL BUT INVISIBLE — a dual-SSOT where the richer
source has no consumer.

## Approach
`phase_model render --project X` already returns `[{id,name,order,status}]` per phase (verified). Wire it
into the resume program so the manifest ladder is surfaced on context reconstruction:
- `code-dev-state-resume.md`: call `TOOL(phase-model, render, --project ...)` and render a one-line ladder
  (per-phase status glyphs: done ✓ · active ◀ · pending ⏳) alongside the existing `_meta.phase`.
- Keep `_meta.phase` as the authoritative "current phase" pointer — this ADDS the manifest view, it does
  not replace it (no behavior change to the gate or the writers).

## Files
- `workspace/programs/code-dev-state-resume.md` (consume + render the manifest ladder)
- `tests/test_phase_ladder_advances.py` (or a new test): assert the consumer reads the manifest + a tool-level
  behavioral check that render reflects an advanced phase.

## Acceptance
- `code-dev-state-resume.md` calls `phase-model render` (the manifest is now a consumed source, not just written).
- Behavioral: advance `study` via `phase-model done --best-effort`, then `render` shows `study:done` — the
  data the consumer now reads carries the advancement.
- Full crucible green; `_meta.phase` semantics unchanged (no dual-SSOT regression).

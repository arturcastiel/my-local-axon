## Summary
S1: the documented risk-tiered git governance (destructive ops human-only unless explicitly
delegated; kernel floor never delegable) has zero mechanical backing at the one chokepoint that
matters — `tools/shell.py gate_check()` (live via the PreToolUse hook). `autonomous_mode.authorized()`
implements the model correctly but has no production callers. This PR wires it in.

## Changes Required
### tools/shell.py
- **What:** Add `_classify_destructive_git(tokens) -> op | None` — per command segment with head
  `git` (robust subcommand extraction skipping `-C <dir>` / `-c k=v` / `--git-dir=...` globals):
  `push` + (`--force`|`-f`|`--force-with-lease`|`--force-if-includes`|`+refspec`) → `force-push`;
  `push --delete`|`push origin :ref` → `branch-delete`; `reset --hard` → `reset-hard`;
  `branch` + (`-D`|`-d`|`--delete`) → `branch-delete`; `commit --amend` → `amend`;
  `rebase` (any form) → `rebase`.
- **How:** In `gate_check()`, when a destructive op is classified: resolve the target repo slug
  (the segment's `-C` dir else cwd → `git config --get remote.origin.url`, parsed for
  `owner/name` from both ssh and https forms; unresolvable → fail-closed deny), resolve my-axon
  via `_axon_paths`, call `autonomous_mode.authorized(myaxon, op, repo)`. Not authorized →
  `GateBlock(reason, code="destructive-git")`. Audit row records op + repo.
- **Why:** Closes the exact S1 hole: `git push --force` currently passes with exit 0.

### tools/shell.py + tools/hooks/enforce_pretooluse.py (kernel floor, unattended scope)
- **What:** When `autonomous_mode.run_active()` (grant mode == "unattended"): deny any write
  (Write/Edit/Bash write-shape) whose resolved target matches kernel-floor patterns
  (`axon/KERNEL*`, `axon/BOOT*`, `axon/core/*`) REGARDLESS of dev-mode. Interactive sessions keep
  current dev-mode semantics (owner present = the human path; per-change confirm by protocol).
- **How:** Shared `_kernel_floor_hit(paths)` helper in shell.py; hook consults it next to the R9
  check; gate_check applies it to inspected paths. Code `kernel-floor`.
- **Why:** Mechanizes the inviolable floor for the only scenario where no human is watching,
  without breaking owner-attended kernel work.

### tests/test_shell_destructive_git.py (new)
- **What:** Classifier unit tests (all 5 ops + negatives: plain push, reset --soft, branch list,
  fetch, -C handling); gate tests with a tmp grant file — empty destructive → BLOCK force-push /
  reset-hard / branch -D / amend / rebase; `destructive:["force-push"]` → force-push allowed,
  reset-hard still blocked; repo-scope mismatch → blocked; unresolvable repo → blocked;
  unattended kernel-floor deny vs interactive allow (dev-mode on).
- **How:** Monkeypatch `_resolve_myaxon`/grant path + a tmp git repo with a set origin.
- **Why:** Core Rule 13 / tests-with-neurons.

## Architecture Impact
gate_check gains its first cross-module governance dependency (autonomous_mode import is lazy,
inside the destructive branch, keeping the hot path stdlib-only and fast). The PreToolUse hook
behavior changes ONLY for destructive git + unattended-kernel writes — every currently-legitimate
call site (workspace-backup push, ordinary commits) is untouched. Ordinary-op grant governance
(plain push/commit) intentionally stays protocol-level (documented), not gate-level — gating it
would break my-axon backup pushes whose repo differs from the grant scope; recorded as a known
limitation in the PR body and the S1 doc-honesty line lands in PR-030.

## Risks & Gotchas
- ⚠ `+refspec` force syntax and `--force-with-lease` must both classify as force-push.
- ⚠ Repo-slug parsing must handle ssh (`git@host:o/n.git`), https, and trailing-slash forms.
- ⚠ The gate executes a read-only `git config` subprocess ONLY on the destructive branch (rare) — keep it out of the common path.
- ⚠ Do not import autonomous_mode at module top (hook latency + circular-import risk).

# PR-001 — hr-team-selector neuron — Layer-1 roster/mode/protocol/tier selection + validation (ACTIVE + router-style test)
Phase 2 plan detail · size M

## Goal
Author workspace/programs/hr-team-selector.md: the Layer-1 SELECTOR neuron that, given a task + invocation mode (M1/M2/M3) + optional filters/roster/mode/protocol/size, validates the request and emits the chosen council CONFIGURATION (roster + per-seat mode + protocol + tier) plus warnings/refusals — WITHOUT calling any seat. It is the entrypoint of the hr-team program family (ADR-002 3-layer split) and the first PR of the bottom-up build (ADR-009). Lands status:ACTIVE with a static router-style contract test (tests/test_hr_team_selector.py) in the SAME change (Core Rule 13 / R_NEW_NEEDS_TEST, ADR-008/D8). It encodes the HANDOFF §4.1 SELECTOR contract: slug-existence, budget, safety, circular-call, context-overflow, no-profession-match, catalog-incomplete checks; mandatory Challenger seat on every roster (all tiers, §16.3); M3 warn-not-fix (validate explicit roster, emit warnings/refusals, never silently repair); LOW_DIVERSITY warning on same model_variant; uniform 1/N seat weights; and advisory_only:true on its output object.

## Risks
- ASSET-PORT ORDERING: the selector READs the catalog registry at workspace/hr-team/catalog/professions/_REGISTRY.md, but the asset port is PR-007 (ADR-009 bottom-up). At PR-001 time the registry file does not yet exist in the repo. Mitigation: (a) spec the read path + the catalog-incomplete refusal so the neuron degrades gracefully when the registry is absent; (b) the static test must NOT depend on the registry existing — it asserts over the .md text only; (c) flag in the PR that end-to-end slug validation is only exercisable after PR-007, OR consider pulling a minimal _REGISTRY.md (slug list only) forward into PR-001 as a test fixture. RESOLVE in plan: either reorder (port registry-only before selector) or accept the graceful-degradation path. This is a real cross-PR dependency to surface to the owner, not silently paper over.
- CIRCULAR-CALL vs FIXTURE MODE: §4.1 blocks recursive self-invocation 'except inside an explicit regression fixture'. The selector must read a fixture-mode flag; if PR-001 invents W:hr-team-fixture-mode, the deliberator/convener fixture (ADR-005 'record ONE real run') must agree on the same flag name — coordinate the key across PR-001/002/003 to avoid drift.
- ROLE CHOICE 'composer': synapse role taxonomy is mostly mutator/reader/gate; 'composer' has one precedent (workflow-run.md). If synapse-validate or registry-drift tests reject 'composer' for a program neuron, fall back to 'reader' (the selector reads catalog + emits a scratch-key config, performs no external mutation). Verify against tests/test_synapse_validate.py before finalizing.
- MODE-PER-SEAT SCOPE: §7 modes can be a preset OR a 6-tuple. Over-specifying per-seat 6-tuples in the selector risks bloating the neuron beyond the convener's actual need (§9.2 only needs mode+protocol+format strings). Keep the selector emitting mode NAMES/tuples as data; let the convener (PR-002) resolve them to prompt fragments — preserve the Layer-1/Layer-2 boundary (§4 mandate).
- REFUSAL-SCRIPT DRIFT: the §11.1 refusal scripts are long verbatim blocks. Embedding all 7 verbatim in the neuron is faithful but heavy; a static test asserting each trigger NAME is robust, but asserting full script text would be brittle. Test on trigger names + key phrases, not whole blocks.
- TIER FLOOR DETECTION: §16.7 'irreversible/multi-domain/release-blocker/customer-facing' detection is keyword-heuristic and inherently fuzzy in a markdown neuron; the floor is a SHOULD that can misfire. Spec it as a documented heuristic with the override path (--size), and test only that the floor LOGIC is present, not its accuracy.

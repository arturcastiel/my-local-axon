## Risks
- H1-fix regex over-reach: if applied outside the leading frontmatter block it could demote a legitimate body H1. Mitigation: scope the substitution to the first ---...--- block only; the test_every_row_has_clean_h1 assertion (title==name) is the backstop. ALTERNATIVE fix considered (prepend '# {Name}' as line 1) rejected because it duplicates the existing bottom H1 and changes more bytes per file.
- Silent cross-link loss: a copy that drops/renames a row would orphan inbound slugs. Mitigation: test_crosslink_resolver pins the exact resolved-edge count and reports dangling slugs; do the copy with cp -a / shutil.copytree (no rename) and verify counts before rewrite.
- DOC-INDEX diff size: this PR adds ~222 lines to workspace/DOC-INDEX.md, making the diff large; that is expected and gate-required, not a defect — note it in the PR body so review isn't surprised. Splitting 7a/7b halves each diff.
- Stale checksums.sha256: copying it verbatim ships a checksum file that no longer validates the (path-rewritten) tree. Mitigation: omit it, or regenerate over the ported tree as the final step.
- programs_registry/dispatch-index 'no-op but run anyway' could surface UNRELATED drift from PR1-6 neurons landing first; that drift belongs to those PRs. Mitigation: if generate/rebuild produce non-empty diffs, confirm they are attributable to the hr-team* neurons (expected) and not the assets, before committing.
- Assets are inert until PR1-5 read them (ADR-009 liveness 'feature goes missing' trap is the inverse concern); since this PR lands AFTER the neuron PRs in the bottom-up order, the readers already exist — but if the orchestrator reorders, ensure the consuming neurons reference workspace/hr-team/ paths that this PR actually creates.

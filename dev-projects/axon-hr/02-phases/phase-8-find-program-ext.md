# PR-008 — find-program: fold # dispatch-phrases into the scoring corpus (repo-wide routing-vocabulary unification)
**Parent-plan**: [02-plan.md](../02-plan.md)
Phase 2 plan detail · size S

## Goal
Extend the find-program neuron so its match corpus ALSO includes each program's `# dispatch-phrases:` line — the same opt-in routing vocabulary smart-dispatch already indexes (dispatch_index._PHRASES) and in-mode routing reads — so a query phrased in natural user vocabulary (e.g. "close out this task") surfaces the right program in find-program too, not only in smart-dispatch. This closes the latent footgun ADR-007 names: routing vocabulary currently must be duplicated into name/desc/PURPOSE for find-program to see it (ADR-007 rejected that local workaround explicitly). Ships as its OWN PR with its OWN test and review, fully decoupled from the axon-hr build (ADR-007 + D6: "SEPARATE small PR, NOT bundled with the hr-team build, keeps reviews decoupled").

## Changes
 1. workspace/programs/find-program.md — Step 2 EXTRACT block (currently lines 49-52: name/desc/purpose/tools): add ONE line right after the `tools` extract — `phrases  ← EXTRACT(W:content, "# dispatch-phrases:") | ""` — using the exact same single-line extract idiom as the adjacent `# desc:` line. This mirrors dispatch_index._PHRASES (`^#\s*dispatch-phrases:\s*(.+)$`): the line is single, `·`-separated free text; treat the whole captured string as one opaque corpus field (no splitting — find-program scores by substring containment, so the `·` separators are harmless).
 2. workspace/programs/find-program.md — Step 2 scoring block (currently lines 56-59): add ONE branch `IF query ∈ phrases → score ← score + 25` immediately after the `desc` branch (keep relative order name>phrases=desc>purpose>tools). Weight rationale to put in an inline comment: a dispatch-phrase is the description a user would actually TYPE (deliberately-authored intent vocabulary), so it earns desc-tier weight (25) — first-class with desc, above PURPOSE(15)/tools(10), at/below name(30). A program with NO `# dispatch-phrases:` line yields `phrases=""`, the branch never fires, and its score is byte-for-byte unchanged — this is what guarantees zero ranking regression for the 145 of 168 programs that have no phrases line.
 3. workspace/programs/find-program.md — PURPOSE section (lines 19-22): update the one sentence that enumerates the corpus — change "Searches program names, `# desc:` fields, PURPOSE sections, and tool references." to also list `# dispatch-phrases:` routing vocabulary, so the neuron's self-description stays truthful to its scoring. One-line doc edit, no behavior.
 4. tests/test_find_program_phrases.py (NEW) — because find-program is a MARKDOWN NEURON (the agent executes its substring scoring by reading the .md; there is no Python tool that runs it), the regression test ships a tiny FAITHFUL Python reference re-implementation of the neuron's Step-2 scorer — name+30 / desc+25 / phrases+25 / purpose+15 / tools+10, substring `query.lower() in field.lower()` — and runs it over the REAL `workspace/programs/*.md` (parsed with the same `# PROGRAM:`/`# desc:`/`## PURPOSE`/`# dispatch-phrases:` extracts the neuron uses). Test 1 (anti-regression PIN): for a handful of existing queries whose intent overlaps name/desc (e.g. query="backup"->workspace-backup, query="dependencies"->deps, query="dispatch"->find-program/smart-dispatch family), assert the top-N (N=5) result LIST equals a hardcoded expected list captured from the scorer WITH phrases OFF — i.e. the test computes both phrases-OFF and phrases-ON rankings and asserts the phrases-ON top-N for these desc-driven queries is identical to phrases-OFF (no displacement). Test 2 (the new win): pick a query that appears ONLY in a program's `# dispatch-phrases:` line and NOT in its name/desc/PURPOSE/tools (e.g. "close out this task" -> code-dev-finalize, or "map the code structure" -> axon-graph), assert it scores 0 under phrases-OFF and ranks #1 (or >=top-3) under phrases-ON. Test 3 (mechanism pin): assert the find-program.md SOURCE now contains both the `EXTRACT(... # dispatch-phrases:)` line and the `IF query ∈ phrases` scoring branch (guards against a future edit silently dropping the corpus field).
 5. Run `python3 tools/dispatch_index.py --workspace workspace rebuild` is NOT required by this PR (this PR only touches find-program, not the dispatch index) — but note in the PR body that smart-dispatch already consumes phrases, so this PR brings find-program to PARITY, it does not change smart-dispatch.

## Source grounding (line-level contracts)
- find-program is a MARKDOWN NEURON, not a Python tool: workspace/programs/find-program.md Step 2 (lines 47-72) is the agent-executed scorer. Corpus today = name(+30, L56) / desc(+25, L57) / purpose(+15, L58) / tools(+10, L59), each scored by substring containment `IF query ∈ X`. There is NO Python implementation of find-program scoring anywhere in tools/ (grep confirmed), so the regression test must ship its own faithful reference scorer.
- The EXTRACT idiom is single-line: `desc ← EXTRACT(W:content, "# desc:")` (L50). The new phrases extract uses the identical pattern keyed on `# dispatch-phrases:` — matching dispatch_index._PHRASES = `re.compile(r'^#\s*dispatch-phrases:\s*(.+)$', re.MULTILINE)` (dispatch_index.py L41).
- dispatch-phrases is ALREADY the canonical routing-vocabulary mechanism: dispatch_index.source_programs() folds it into the index entry as `phrases` (L77-80, joined by ' · '), and dispatch.py match builds its corpus as `f"{p} {description} {phrases}"` (L236-239). So find-program reading the same `# dispatch-phrases:` line is literal repo-wide UNIFICATION, not a new convention. The phrases field is documented there as 'never displayed' routing vocabulary — find-program likewise should score on it but the OUTPUT still shows only name+desc (find-program.md L86-91 prints m.name/m.desc only), so no display change is needed.
- Phrase line is a single `·`-separated string (verified: code-dev-finalize.md = 'close out this task · finalize the work · wrap up and hand off · finish the project'); 23 of 168 programs carry one. Substring scoring over the whole string is correct — the `·` separators don't interfere with `query in phrases`.
- ADR-007 (_decisions.md L101-115): the decision is literally 'EXTEND find-program with a one-line EXTRACT so it also reads # dispatch-phrases', 'Ships as a SEPARATE small PR with its own test, NOT bundled with the hr-team build', and the Note: 'find-program is shared infra -> the extension PR must not regress existing ranking (test pins current top-results + the new phrase hits)'. This spec's two tests are exactly that pin.
- ADR-009 (_decisions.md L134-145) lists this as 'PR-X (separate) find-program dispatch-phrases extension (ADR-007)' OUTSIDE the bottom-up PR1-PR7 chain, confirming depends_on=[] (no hr-team dependency). Core Rule 13 / R_NEW_NEEDS_TEST: neuron edit + test in the same PR.
- find-program is an EXISTING program (workspace/programs/REGISTRY.json L1228, tools/REGISTRY.json, dispatch-index.json all already list it). It is an EDIT not an ADD, so per the gate-verdict memory the NEW-tool obligations (onboarding-count / liveness / keystone / freshness deltas) do NOT apply here — only contract-test parity does. contract-version stays neuron-contract v1.1 (additive, non-breaking change to corpus).
- Existing precedent for the test shape: tests/test_dispatch_graph_routing.py pins 'unambiguous intents top-rank' (==1) and 'ambiguous intents surface top-3' (<=3) and asserts controls are not regressed (test_controls_not_regressed) — the same honest top-1/top-3 split this spec adopts for the phrase-only query.

## Risks
- The regression test relies on a Python REFERENCE re-implementation of the neuron's scorer, which can drift from what the agent actually executes when reading the .md. Mitigation: keep the reference scorer dead-simple and a 1:1 transcription of the four (now five) `IF query ∈ X -> +W` lines, and add test_phrases_extract_and_score_present_in_source so the .md and the reference can't silently diverge on the mechanism.
- Choosing control queries that are genuinely desc/name-driven matters: if a control query happens to also appear in some program's dispatch-phrases, phrases-ON could legitimately differ from phrases-OFF and the no-regression assertion would (correctly) fail. Mitigation: pick control queries verified to have zero phrase overlap, or assert the WEAKER but still-valid invariant that no program's RELATIVE order among the prior top-N changes (phrases can only ADD score, never subtract, so a phrase-less control program can only be displaced if a phrase-bearing program out-scores it — select controls where that can't happen).
- Weight choice (+25, == desc) is a judgement call; if a reviewer prefers phrases below desc, change the single constant. The no-regression guarantee holds for ANY positive weight because phrase-less programs are untouched; the weight only affects how strongly phrase-bearing programs rank, which the phrase-only test covers at top-1/top-3 granularity (robust to the exact constant).
- Substring (containment) scoring means a long dispatch-phrases line increases the chance a short generic query substring-matches incidentally — but this is the SAME containment semantics the existing desc/purpose/tools branches already use, so it introduces no new class of false-positive beyond what authors opted into by writing the phrase.

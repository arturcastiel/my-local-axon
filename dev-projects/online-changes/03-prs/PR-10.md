> Phase 3 · effort M · depends: PR-9 (tools/retrieval_index.py with the dense index, the `query` subcommand, `dump-case`, and the shared sparse()/dense() functions + RetrievedChunk dict shape). PR-10 imports sparse()/dense() and extends query; it cannot land or be graded before PR-9's dense list and dump-case fixtures exist. ## Objective
Add a pure-stdlib, model-free reciprocal-rank-fusion merge of the existing sparse (TF-IDF) and dense (PR-9) retriever outputs, exposed as `retrieval_index query --hybrid`, so a chunk that is rank-1 in one retriever and rank-3 in the other beats a high-scoring-but-irrelevant chunk. This fixes the vocabulary-mismatch mis-routes that dispatch's PREFER_COMPILED_FLOOR exists to paper over, and flips rag_maturity_audit B3 (0/1->2) — but it lands ONLY because a retrieval_eval metric delta proves precision rose without recall/latency regression, not because the audit's "reciprocal rank fusion" keyword is present.

## Approach
RRF is ~20 lines: for each ranked list, score[id] += 1/(k+rank); k=60 is the canonical TREC constant and is exposed as a default arg so the test can pin determinism. The merge is on chunk `id` (the field retrieval_eval keys precision/recall on at retrieval_eval.py:73,81), so a fused list scores through retrieval_eval.score_case unchanged — zero new metric code. The wiring point is the `query` subcommand PR-9 adds to retrieval_index.py: --hybrid simply fans the same query to PR-9's sparse() (lifted from dispatch.py:85 tfidf_similarity / agent_memory.py:154 _tfidf_rank) and dense(), then fuses. Because both inputs are already RetrievedChunk dicts, fusion needs no knowledge of either retriever. Latency is carried as max-per-id so retrieval_eval.latency_ms (retrieval_eval.py:115-120, which prefers explicit per-chunk latency) caps the p95 budget honestly. The fixture is produced by dump-case over the LIVE corpus (the anti-mirage discipline the repo's recent 'proof the mirage lacked' commits enforce), so the precision lift is measured on real retrieval, not a hand-tuned toy.

## Gate (what proves it landed — metric/behavior, not keyword)
tests/test_retrieval_fusion.py::test_rrf_lifts_precision_without_recall_or_latency_regression (also the metrics_manifest tripwire) PROVES it landed by behavior: on the hybrid_rrf.json fixture (relevant chunk = rank-1 dense / rank-3 sparse; plus a high-sparse irrelevant chunk), retrieval_eval.context_precision(rrf(sparse,dense)) is strictly greater than context_precision(sparse_only) and >= context_precision(dense_only), AND context_recall does not drop below max(single-retriever recall), AND latency_ms_p95 of the fused result does not exceed the union's max single-retriever latency. This is the Master-plan Pillar-3 exit ('hybrid improves precision without regressing recall or latency'). It is NOT the rag_maturity_audit.py:226 B3 keyword (_contains_any 'reciprocal rank fusion'/' rrf') — that flips green incidentally from the docstring, but a planted string with no real fusion would FAIL this test, so the keyword is necessary-not-sufficient and the metric is the true lock.

## Risk
Low-medium. (1) rrf math is trivial and stdlib-only, so the function itself is near-zero risk; the real risk is the fixture: if the live dump-case does not actually surface a rank-1-dense/rank-3-sparse relevant chunk, the precision-lift assertion has nothing to prove — mitigate by selecting a corpus query with known vocabulary mismatch (the dispatch mis-route class) and verifying the split-rank condition holds in the dumped fixture before committing it. (2) Keyword-theatre trap: B3 flips green from the docstring alone; mitigated by making the metrics_manifest tripwire the lock so metric_integrity.py audit fails if the behavioral test is removed. (3) Latency: RRF runs BOTH retrievers, so hybrid p95 is naturally higher than either single path — the gate caps it at the union max (acceptable, not a regression vs running both) but --hybrid stays opt-in so always-on paths keep their single-retriever latency.

## Proportionality (why this scope)
Right-sized: ONE new library file (~40 lines) + ONE flag on an existing subcommand, no new CLI tool — honoring the REDUCE-tool-surface mandate by folding fusion under retrieval_index's query rather than shipping a standalone `retrieval_fusion` executable (the honest-callout's explicit guidance to prefer subcommands over the ~6-8 proposed new tools). Smaller would mean no real fusion (just the audit keyword = mirage). Larger (wiring --hybrid into dispatch/agent_memory/library at once, or adding min-max calibration as the default) is out of scope: each consumer opt-in is its own metric-locked PR, and calibrated fusion is shipped as a documented-but-ungated alternative so this PR proves exactly one claim. Net new surface: 1 lib + 1 fixture + 1 test + 1 manifest row; net new user-facing capability: 1 read-only flag.

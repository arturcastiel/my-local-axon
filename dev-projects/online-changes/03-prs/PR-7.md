## Objective
Generalize the existing claim-confidence gate (library.partition_claims, library.py:174-193) onto chunk score so low-relevance chunks are dropped before context assembly (flips audit D2), and add a token-budget packer to context.py that dedups + greedily fills the host-model budget with a per-source cap (flips audit D3). All-reject is surfaced as the local-retrieval-failed signal that is the principled web-escalation trigger. No new dense substrate; both surfaces already exist and are already in the audit's SEARCH_GLOBS.

## Approach
D2 (noise gate): The accept/qualify/reject pattern is already implemented in library.partition_claims (library.py:174-193) over {claim,confidence}. partition_chunks is a near-identical sibling over {id,text,score}: accept>=accept_thr, ambiguous in [reject_thr,accept_thr), reject<reject_thr. Scores already exist on every live surface (retrieval_eval.RetrievedChunk.score, dispatch confidence at dispatch.py:240-242, agent_memory.recall ranked score) and in the fixtures — no new scoring substrate. The gate returns all_reject (no chunk clears reject_thr) which is the CRAG corrective-retrieval / web-escalation trigger named in the D2 leverage row. Placing it in library.py (not a new tools/retrieval_quality.py) is deliberate: library.py IS in rag_maturity_audit SEARCH_GLOBS (line 28) and a new file would be invisible to the audit scan, so reuse here both REDUCES surface and makes the D2 vocabulary land on a real, tested function. D3 (packing): context.py today only ESTIMATES tokens and resolves the host-model budget (resolve_limit, line 95-126) — there is no assembler. pack() dedups by chunk id (stops double-counting the same chunk against pressure), greedily fills the resolve_limit() budget using the existing _count_tokens (line 36-45), and caps per source-path for diversity. context.py is also already in SEARCH_GLOBS (line 29). Both functions are pure/deterministic (model-free) and READ-ONLY: the gate is stdin->stdout, the packer never calls write_state. CLI surface is added as subcommands of the existing library.py / context.py CLIs (REDUCE: no new standalone tools), mirroring cmd_partition (library.py:254-257) and the existing context.py action dispatch (line 176-245).

## Risk
Low. Two small pure functions + thin CLI wiring on two files that are already tested and already scanned by the audit. Main subtlety: choosing accept/reject thresholds (default accept=0.50, reject=0.20) so the 0.11 fixture chunk lands in reject while the 0.94/0.98 chunks stay accept — pinned by the fixture-driven test, not guessed. Second subtlety: token_iou must be invariant under packing at the tight budget chosen by the test, so the test sets token_limit high enough to retain both relevant chunks (packing drops only the reject/duplicate, never gold) — asserted explicitly. _count_tokens has a tiktoken/fallback branch; pack() is correct under both since the test asserts <=limit and id-dedup, not an exact token count.

## Proportionality (why this scope)

Not bigger: the live retrieval_trace.py emitter, rerank.py (D1), and web-escalation wiring (F-rows) are deliberately separate PRs — this PR only produces the gate buckets + packed context they will later consume.

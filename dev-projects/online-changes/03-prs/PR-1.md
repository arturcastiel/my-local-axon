# PR-1 — Guarded MCP exposure of retrieval-eval + rag-maturity-audit (read-only; closes the --append-log/--root write hole)

> Phase 0 · effort S · depends: None. Stands alone on main. REGISTRY.json already has both tools ACTIVE (verified), and the _is_readonly_call/exposed_tools/handle_call machinery already exists.

**Status:** ✅ MERGED — a3cdd06, gate GREEN. Write hole closed; both tools MCP-exposed read-only. Pushed to origin/main.

## Objective
Expose the two deterministic read-only RAG tools (retrieval-eval, rag-maturity-audit) to every MCP client through tools/mcp_server.py's allowlist, and in the SAME diff close the verified write/retarget hole: today _is_readonly_call passes ["evaluate","--append-log"] (a remote client could mkdir+append JSONL under private my-axon/) and ["audit","--root","/etc"] (retargets the maturity scan). One file's allowlist triad unlocks the entire non-shell harness tier (Claude Desktop, Copilot-as-MCP-client, external agents) with no new code, no model dependency, and no per-harness wiring, while keeping both tools strictly read-only-by-default.

Reuse the existing _R stub + monkeypatch.setattr(srv.subprocess,"run",...) pattern from test_handle_call_refuses_arbitrary_write.

## Approach
Single diff in tools/mcp_server.py touching the three-part allowlist (SAFE_TOOLS + READONLY_SUBCOMMANDS + _FORBIDDEN_FLAG_TOKENS) that _is_readonly_call (line 67) and exposed_tools (line 86) already enforce — no new functions, no new code paths. Order matters and is enforced by review: the two flag-denies and the subcommand pins land in the SAME commit as the SAFE_TOOLS additions, so exposure is never granted without the guard. Mechanism is unchanged: handle_call (line 129) gates every call through _is_readonly_call BEFORE building the subprocess argv (line 146), so a forbidden flag returns isError with no subprocess at all — exactly how the existing --out/--workspace denies work. Do NOT route through aegis: these are deterministic, model-free query reads (retrieval_eval.evaluate over fixtures; rag_maturity_audit.audit over the repo corpus) — read-only-by-default is the correct contract, and aegis would add a write/approval gate that does not apply. No standalone tool is added: we reuse the existing subcommand-gated subprocess bridge rather than expanding surface. liveness.py already regexes mcp_server.py for the quoted tool name (reached_by, the mcp branch: re.search(rf'[\"\\']{re.escape(name)}[\"\\']', surfaces[\"mcp\"])), so both tools auto-register their MCP reach with zero extra wiring.

## Gate (what proves it landed — metric/behavior, not keyword)
Behavioral, not a keyword: in the crucible/pytest run, (a) the new test_handle_call_refuses_retrieval_eval_append_log proves handle_call(retrieval-eval,["evaluate","--append-log"]) returns isError AND the monkeypatched subprocess.run is never called (the ran flag stays False) — i.e. NO file is created under my-axon/log/retrieval-evals/; (b) test_handle_call_refuses_audit_root proves --root is refused before any subprocess; (c) the existing test_integration_initialize_and_list_via_client drives the REAL server through mcp_client and now lists both new tools while still asserting names <= srv.SAFE_TOOLS. Manual one-shot proof: `python3 -c "import sys;sys.path.insert(0,'tools');import mcp_server as m;print(m._is_readonly_call('retrieval-eval',['evaluate','--append-log']));print(m._is_readonly_call('rag-maturity-audit',['audit','--root','/etc']))"` must flip from today's (True,'') / (True,'') to (False, ...) / (False, ...).

## Rollback
Pure allowlist data edit in one file plus additive tests; revert the single commit. Removing "retrieval-eval"/"rag-maturity-audit" from SAFE_TOOLS instantly un-exposes them (exposed_tools fail-safe filters unknown names), and dropping the two tokens from _FORBIDDEN_FLAG_TOKENS restores prior behavior. No schema, no migration, no state, no consumer depends on the new exposure. Zero blast radius beyond MCP tool visibility.

## Risk
Low. The only real risk is shipping exposure without the deny — explicitly mitigated by requiring both flag-denies + subcommand pins in the same commit and locking them with the two refusal tests (gate fails if either deny is missing). Secondary: the generic `args` passthrough means a future-added mutating subcommand to either tool would be reachable — mitigated by the READONLY_SUBCOMMANDS pin (only "evaluate"/"audit" positional is legal, any other is refused). Both tools are model-free and deterministic so no nondeterminism enters the MCP surface.

## Proportionality (why this scope)
Smallest edit that closes a live write hole AND grants the exposure — ~5 verified data lines in one file plus 3 additive tests. NOT bigger: rejected adding a standalone MCP tool, a list-tools/registry rework, or an in-process import shim (the avail.json honest_callouts flag those as lower-reach polish or speculative; Phase-3 in-process API has no consumer yet). It REDUCES net surface by reusing the existing subcommand-gated subprocess bridge instead of new code, and tightens the read-only contract (two new denies) rather than loosening it. NOT smaller: exposing without the --append-log/--root denies would silently break read-only-by-default by letting a remote client mkdir+append under private my-axon/ or retarget the scan — so the guard cannot be deferred to a follow-up. Single chokepoint (the allowlist triad) reachable identically by every MCP client.

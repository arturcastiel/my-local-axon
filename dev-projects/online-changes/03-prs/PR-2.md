**Status:** ✅ MERGED — 2a249be, gate GREEN. Registry self-describing; RAG tools in menu. Pushed to origin/main.

## Objective
list-tools renders a blank purpose for every tool absent from its 11-entry hardcoded dict — that is ~142 of 153 registered tools (verified: REGISTRY.json has 153 tools, all 153 carry a populated `purpose`, the dict at list-tools.md:23-35 names only 11). One-character-class edit makes the program fall back to the registry's single source of truth, so the whole registry becomes self-describing; two quiet menu doc lines make rag-maturity-audit and retrieval-eval findable without adding a numbered mode or any executable surface.

Prevents a stale-mirror RED in test_compiled_quality_ratchet / test_compiled_regression; keep the `# COMPILED:` / `Source:` / `Tokens:` header (asserted by test_compiled_freshness.py).
- `workspace/programs/menu.md` — after line 334 (`cron breaker-status …`), inside the existing SELF-OBSERVE band (header at line 326), add two plain doc lines: `→ "     retrieval-eval evaluate   — RAG retrieval-quality metrics over fixtures"` and `→ "     rag-maturity-audit audit  — score retrieval/grounding/eval readiness"`. Quiet band, NOT a numbered [N] mode; no new TOOL()/dispatch surface.

## Approach
Grounded in real symbols. (1) list-tools.md:45 today is `→ "  {status-icon}  {t.name:<20} {purposes[t.name]}"` where `purposes` is the hardcoded dict at lines 23-35 (clock, calculator, tokenizer, diff, validator, notify, kv-store, semantic-search, document-parser, web-search, translate — 11 keys) and `registry ← RETRIEVE(W:tool-registry)` (line 20) binds `t` with `.name`, `.status`, and `.purpose` per entry (verified every REGISTRY.json entry has script/status/category/purpose). The render-loop status-icon line at 44 already reads `t.status`, proving `t` exposes registry fields, so `t.purpose` is in scope. Change `{purposes[t.name]}` → `{purposes[t.name] | t.purpose}`: the `|` interpolation-default returns the registry purpose when the dict has no key. Precedent for this exact interpolation-default form is live at menu.md:360 (`confidence: {c.score}` line uses `{c.reason | '—'}`), so this is established AXON syntax, not an invented API. (2) Recompile: compiled/list-tools.cmp.md line 24 carries the stale `{purposes[t.name]}`; compilation is cognitive (no mechanical refresh) so a stale mirror would drift and, under prefer-compiled dispatch, serve the old blank-purpose render — exactly the failure class PR-W2 retired a mirror for. Run `python3 tools/compile.py format` (the documented format subcommand, verified via --help) targeting list-tools, keeping the header invariants. (3) menu.md SELF-OBSERVE band (lines 326-334) already lists self-measurement surfaces (orchestrator, dispatch-stats, drift check, cron breaker-status) as quiet one-liners; append the two RAG tools in the same shape after line 334. Per the study's honest_callout #3 and the brief: the REGISTRY `description` field is read by NO live surface (find-program reads `# desc:`, MCP/CLI read `purpose`), so this PR does NOT fill descriptions — the substantive fix is the fallback. Read-only/model-free/determinism preserved: both files are `!NORM | read-only` render programs, no new executable surface, no MCP allowlist change.

## Gate (what proves it landed — metric/behavior, not keyword)
A behavior assertion, not a keyword. New test (tests/test_list_tools_registry_fallback.py) parses workspace/programs/list-tools.md, asserts line 45's render contains the `{purposes[t.name] | t.purpose}` fallback (string-level, on the source render line) AND — the load-bearing check — picks a tool present in REGISTRY.json but ABSENT from the hardcoded dict (e.g. retrieval-eval / rag-maturity-audit / coverage-gate), and asserts the resolution rule yields that tool's non-empty registry `purpose` (i.e. dict-miss → registry purpose, not blank). This proves the ~142 blanked tools now render a purpose. A second assertion pins the menu: parse menu.md, assert both `retrieval-eval evaluate` and `rag-maturity-audit audit` lines exist within the SELF-OBSERVE band (between the `SELF-OBSERVE` header line and the following `META TOOLS` header) — band-scoped, so a stray mention elsewhere can't satisfy it. The crucible gate must stay passed:true with test_compiled_quality_ratchet (RED_BASELINE=10) and test_compiled_freshness green, proving the recompiled mirror is not stale and not orphaned.

## Risk
Low. (1) `t.purpose` is guaranteed populated (verified all 153 REGISTRY entries have it), so the fallback never re-blanks; the `|` short-circuits only on dict-miss, so the 11 curated overrides are untouched. (2) Recompile is the only step with a footgun — forgetting it leaves a stale mirror that the ratchet test catches at gate time (fail-closed, not a silent regression). (3) Menu lines are pure text in a read-only render; worst case is a cosmetic wording nit. No write path, no MCP surface, no determinism/model dependency introduced.

## Proportionality (why this scope)
Scoped to the substantive fix and nothing more. NOT bigger: the study's honest_callout #3 and the brief both flag that filling REGISTRY `description` fields is ceremony (no live surface reads them), so this PR deliberately skips that and changes only the one render line that actually blanks tools. It adds NO new standalone tool (respecting REDUCE-surface) and NO numbered menu mode (respecting uncluttered-home). NOT smaller: the recompile is mandatory, not optional — omitting it trades a visible blank for a hidden stale-mirror that prefer-compiled dispatch would serve, so it ships in the same commit. Surface-area note: net new executable surface = zero; the only additions are two doc strings and one interpolation-default operator, both using idioms already live in the codebase (menu.md:360). The capability (registry-backed self-description) is locked by a behavior test that resolves a real dict-miss to a real registry purpose — not by a rubric keyword.

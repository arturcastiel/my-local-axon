## Objective
Land a model-free `tools/retrieval_index.py plan` subcommand that expands (paraphrase / step-back / HyDE-lite from REGISTRY descriptions), decomposes (clause/and/`?` split), and routes a raw prompt, emits paired baseline-vs-expanded RetrievalCases, runs the winning variant's subqueries through `retrieval_index query --hybrid` + RRF, and injects that variant into the orchestrator OBSERVE state so every sparse surface stops consuming the bare prompt. This is dimension C's keystone (flips C1/C2/B6) and is the leverage move that lifts recall across dispatch, agent-memory, library-dev and web_search at once with one planner instead of six.

## Approach
Build the planner as a `plan` subcommand of `retrieval_index.py` (NOT a standalone tool — honors the REDUCE-surface mandate; leverage.json honest_callout (c) explicitly says fold planner-class capability into retrieval_index/retrieval_eval subcommands). It is fully model-free and deterministic: `expand_query` paraphrases by stem/synonym over `WORD_RE` tokens, does step-back by dropping the single longest/rarest token, and builds a HyDE-lite pseudo-doc by pulling the `purpose` field from `tools/REGISTRY.json` (and `desc` from the top sparse-matched program) — the same fields `synapse_suggest._load_candidates_from_registry` (synapse_suggest.py:492-515) already reads, so no new corpus. `decompose_query` promotes `library.gap_queries` (library.py:211) clause-extraction rather than reimplementing it. `route_query` is a 6-line rule classifier mirroring the threshold/fallback split precedent in dispatch.py:249. Each variant's subqueries run through PR-9's `query --hybrid` (sparse()+dense()) and are merged by PR-10's `retrieval_fusion.rrf(k=60)`; results are serialized into the exact `RetrievedChunk` dict shape `retrieval_eval._load_case` parses (retrieval_eval.py:50-58), so `score_case` (retrieval_eval.py:123) grades the plan with no new metric code. Injection: the orchestrator OBSERVE block (orchestrator.md:54-70) gains a `retrieval-query` derived from `plan --winner-only` and stamps it into `state.retrieval-query`; the FL-05 fallback dispatch call (orchestrator.md:89) consumes it. `recent-input` stays intact so the anticipation sideband (orchestrator.md:127) is unaffected. The planner is hosted as `s0` in adaptive-free-text.yml (the file B6 already scores 1 on via rag_maturity_audit.py:241) so the expanded query reaches synapse-suggest's `dispatch_tfidf`/`goal_alignment` (synapse_suggest.py:73,223) before ranking. Crucially: `tools/query_planner.py`/`retrieval_index.py` are NOT in the audit's SEARCH_GLOBS (rag_maturity_audit.py:22-45 — only specific tool filenames are scanned), so C1/C2/B6 cannot be flipped by planting strings in the new code; the keyword strings must live in the in-glob `workspace/programs/retrieval-plan.md` + `adaptive-free-text.yml`, and — by design — the row only counts as truly closed when the paired-fixture metric gate below passes in the crucible.

## Gate (what proves it landed — metric/behavior, not keyword)
retrieval_eval metric deltas on the paired fixtures, NOT keyword presence: (1) `rev.context_recall(_load_case(query_plan_expanded.json)) > rev.context_recall(_load_case(query_plan_baseline.json))` for identical gold relevant_ids, and `faithfulness_proxy` does not regress — proves expansion raised recall (C1). (2) On multiquery_compound.json: `rev.context_recall(fused_case) == 1.0` while `max(recall(subquery_i)) < 1.0` and the raw single-query recall ≤ 0.5 — proves decomposition+RRF reaches both gold ids no single query reaches (C2/B6). (3) `ri.route_query(q) == expected_path` for one fixture per path AND the chosen path's retrieval yields the gold id at precision ≥ 0.5 (C3 strengthened). (4) The planner's emitted JSON validates through `rev._load_case` without error and `rev.score_case` returns all 5 metrics — proves the RetrievalCase contract holds. CI/crucible: rag_maturity_audit C1/C2 read 2 and B6 reads 2, AND the metrics_manifest tripwire for query-plan-recall must have a falsifying test (recall(expanded) ≤ recall(baseline) fails the build), so the row cannot pass on the keyword alone.

## Risk
Medium. (1) Keyword-theatre temptation is real and called out in leverage.json honest_callout (b): C1/C2/B6 flip on co-located strings via rag_maturity_audit _contains_any (rag_maturity_audit.py:251,254,240). Mitigated structurally — the new tool code is outside SEARCH_GLOBS, and the row is gated on the paired-fixture recall delta with a falsifying metrics_manifest test, so a string without the metric fails the build. (2) Orchestrator OBSERVE is on the hot path of every adaptive/free-text tick; a malformed `plan` call could stall the loop — mitigated by the `| recent-input` fallback making plan failure a no-op. (3) HyDE-lite reading REGISTRY purpose text could inject noise tokens that LOWER precision on some queries; the gate requires faithfulness_proxy not regress and `--winner-only` picks the baseline-or-expanded variant with higher measured recall, so a bad expansion is discarded, not forced. (4) Determinism: synonym/stem expansion must be a fixed table, not locale-dependent — locked by test_plan_is_deterministic.

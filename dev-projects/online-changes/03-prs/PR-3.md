## Objective
Convert AXON's deterministic retrieval evaluator from a hand-fixtures-only toy into a measurement over live sparse retrieval. Build tools/retrieval_trace.py that captures the (query, retrieved[{id,text,score,latency_ms}], answer, relevant_ids, gold_spans) row that retrieval_eval._load_case already parses, wire capture into dispatch.match (after `scored`) and agent_memory.recall (before return), and finally invoke the built-but-never-called retrieval_eval._append_log via --append-log in workspace/programs/retrieval-eval.md. The captured row IS a RetrievalCase, so retrieval_eval.score_case grades it with zero new metric code. This is the foundation every later wave (including the dense B1 index) is graded against.

## Approach
GROUND: retrieval_eval.py already defines RetrievedChunk(id,text,score,latency_ms) at line 27 and RetrievalCase(id,query,answer,retrieved,relevant_ids,gold_spans) at line 34; _load_case (line 48) parses exactly that JSON; score_case (line 123) returns the 5 metrics; _append_log (line 206) writes MYAXON_ROOT/log/retrieval-evals/<date>.jsonl and is reachable only via the already-declared --append-log flag (line 187/218) that nothing calls today. So the entire 'measurement over live traffic' problem reduces to: emit the JSON _load_case wants, from data the retrievers already compute.

dispatch.match already has everything: corpus[i] (program+desc text, line 228), programs[i] (id), scores[i] (cosine, line 230), and sorted `scored` (line 239). One transform from those produces retrieved=[{id,text,score,latency_ms}]; capture writes it to log/retrieval-traces/<date>.jsonl. agent_memory.recall already returns results=[{id,tier,scope-ref,score,bindings,snippet}] (line 210-216) — snippet is text, id is id, score is score, bindings are the provenance source ids. Same one-line transform, same capture call.

retrieval_trace.replay imports retrieval_eval and calls score_case — ZERO new metric code, honoring model-free determinism and reduce-surface (one new tool, scoring delegated). dump-case writes a captured trace out to tests/fixtures/retrieval_eval/*.json so retrieval_eval.evaluate() grades REAL retrieval, producing the live fixtures every later wave (incl. dense B1) is graded against.

HONESTY ON relevant_ids/precision: at capture time the retriever does not know ground truth, so relevant_ids/gold_spans default empty. With them empty, context_precision returns 0.0 (line 71-74: no chunk.id in empty set) and context_recall returns 1.0 (line 79-80 early-return). Those two are uninformative until a dump-case is later LABELLED with relevant_ids — that labelling is exactly what makes a curated-quality fixture from live traffic. Immediately-meaningful live signals: token_iou (vs gold_spans when present), faithfulness_proxy (answer-token support), latency_ms — plus the captured trace itself for E8/D4. The gate therefore asserts non-zero precision/recall ONLY on a dump-case the test LABELS, never on a raw capture; the truthful reading of the leverage gate 'LIVE dump-case fixtures (not hand-made)'.

AUDIT KEYWORDS ARE A SIDE EFFECT, NOT THE GOAL: the capture comment in dispatch.py co-locates 'retrieval'+'trace' (E8 line 307) and the retrieval-eval.md --append-log wiring co-locates 'append_log'/'jsonl' (E6 line 300) — but the LOCK is the behavior test, not the keyword. Surface cost: +1 tool + 1 test; scoring/metrics reused from retrieval_eval; capture is read-only-append and OFF by default behind a pref so it cannot perturb dispatch routing precision until explicitly enabled.

## Gate (what proves it landed — metric/behavior, not keyword)
tests/test_retrieval_trace.py must prove, by behavior in the crucible, all of: (1) a trace from retrieval_trace.capture() carries every key retrieval_eval._load_case requires (id,query,answer,retrieved[{id,text,score,latency_ms}],relevant_ids,gold_spans) and round-trips: retrieval_trace.replay(trace) == retrieval_eval.score_case(loaded) returning the 5 metrics (context_precision/context_recall/token_iou/faithfulness_proxy/latency_ms). (2) dump-case on a trace LABELLED with relevant_ids+gold_spans, fed through retrieval_eval.evaluate(), yields non-zero context_precision AND context_recall (the 'live dump-case scores real retrieval' proof). (3) --append-log grows MYAXON_ROOT/log/retrieval-evals/<date>.jsonl by exactly len(result['cases']) lines (proves _append_log is now actually invoked). (4) rag_maturity_audit re-run shows E8 row score == 2 (retrieval+trace co-located in tools/dispatch.py) and E6 evidence non-empty — assert via the audit's row scores, NOT by grepping a keyword. `crucible gate` returns passed:true with the new test bound.

Revert by: (a) delete tools/retrieval_trace.py, tests/test_retrieval_trace.py, the REGISTRY.json 'retrieval-trace' entry; (b) revert the 2 guarded capture call-sites in dispatch.py (~line 244) and agent_memory.py (~line 216) — both try/excepted and pref-gated so removal restores byte-identical retriever behavior; (c) revert retrieval-eval.md to 'evaluate --format {format}' (drops --append-log). ## Risk
LOW-MEDIUM. Capture is append-only and OFF by default behind read_pref(ws,'retrieval-trace','off'), so it cannot perturb dispatch routing or memory recall results — the chief risk (a self-inflicted retrieval-precision cost from instrumenting the hot path) is neutralized. Real risks: (1) import cycle — retrieval_trace imports retrieval_eval, dispatch/agent_memory import retrieval_trace; keep retrieval_trace dependency-light (stdlib + retrieval_eval only) and lazy-import it at the call-sites to avoid import-time cost/cycle. (2) Honesty trap: precision/recall are 0.0/1.0 on UNLABELLED captures — mitigated by only asserting non-zero on a labelled dump-case and documenting this in the docstring so no one mistakes a raw capture for a graded result. (3) Date/path: log dir built from MYAXON_ROOT exactly like _append_log:207, inheriting the same env resolution.

## Proportionality (why this scope)
Right-sized: ONE new tool (the prompt's named keystone) plus two ~3-line guarded call-sites and one program-line edit. Surface-area note: tools/ already holds 160+ files and the project is actively reducing tool count, so this PR deliberately (a) adds NO new metric code — replay/dump-case delegate entirely to retrieval_eval.score_case/evaluate; (b) keeps capture/replay/dump-case as subcommands of a single file rather than three tools; (c) invokes the already-built _append_log instead of writing a parallel history. A smaller version (just --append-log) would not make live retrieval gradeable — the adapter is the irreducible unlock. A larger version (folding rerank/quality/drift in now, or building the dense index) is explicitly deferred: those are separate phase-2/3 rows that this adapter must exist FIRST to grade, so bundling them would break gate-first sequencing and conflate proofs.

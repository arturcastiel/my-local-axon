# PR-5 — Deterministic structural chunker as a retrieval_eval subcommand + per-chunk metadata (kill the raw-text[0..8000] truncation)

> Phase 1 · effort M · depends: None — this is the A-dimension keystone and the first PR. It does NOT depend on the retrieval_trace adapter (the synthesis's separate Wave-1 keystone). It precedes any dense work (B1): the dump-case fixtures B1 must be graded against are exactly the chunk corpora this PR produces. The metadata_filters quick-win (B5) and retrieval_trace can build on the {source,section,doc_type} fields this lands, but neither blocks this PR.

**Status:** ✅ MERGED — f27681a, gate GREEN. Audit 46→48/70 (A2 OK). 8000-char truncation killed. Pushed to origin/main.

## Objective
Add a model-free structural chunker that emits {id,text,char_span,source,section,doc_type,hash} per chunk, fold it in as a `retrieval_eval chunk` subcommand (not a new standalone tool), and replace library-dev-ingest's whole-document `raw-text[0..8000]` truncation with that chunker. This is the A-dimension keystone: each emitted chunk's {id,text} is exactly the {id,text} retrieval_eval._load_case already grades against, so it becomes the corpus the relevant_ids/gold_spans of every later A/B row attach to — and it genuinely flips A2 (per-chunk metadata) and improves A1 (structural chunking).

## Approach
Ground truth checked: retrieval_eval.py already dispatches on a positional `command` with `choices=[\"evaluate\"]` (line 184) and `_load_case` (line 48-66) maps `retrieved[].{id,text}` against `relevant_ids`/`gold_spans`. So a chunk's {id,text} is one transform from a RetrievedChunk — folding the chunker here (not a new tools/chunking.py) means a chunked corpus is directly gradeable with ZERO new metric code, and it honors the REDUCE-surface ethos and the honest_callout's explicit push to 'fold proposed new tools into retrieval_eval/retrieval_index subcommands.' Determinism: sha256-derived ids (matching shadow.py:49), pure boundary regex, no embeddings/LLM. doc_type comes from the caller's extension; section comes from the nearest markdown heading. The truncation being killed is the literal `raw-text[0..8000]` at library-dev-ingest.md:182 (the only place the prompt's '0..8000' lives, confirmed by grep). READ-ONLY: the `chunk` subcommand reads --file and prints JSON; it mutates nothing, so retrieval-eval.md stays `read-only`.

## Gate (what proves it landed — metric/behavior, not keyword)
A2/A1 proven by BEHAVIOR, never by the audit keyword. rag_maturity_audit.py:197 flips A2 on `_contains_all((\"chunk\",\"metadata\"))` and A1:192 on strings like \"chunk overlap\" — the honest_callout warns this is keyword theatre, so the audit row is necessary-not-sufficient. The real gate is three pinned assertions in tests/test_retrieval_eval.py: (1) every chunk from chunk_document(chunking_doc.md) carries all 7 keys {id,text,char_span,source,section,doc_type,hash} and ids are unique+stable across two runs (determinism); (2) building a RetrievalCase from those chunks (chunk.{id,text} -> retrieved[], one chunk's id in relevant_ids, its section text as a gold_span) replays through rev.score_case to all 5 metrics with context_precision and token_iou > 0 — proving the chunk shape IS a gradeable corpus; (3) on chunking_doc.md, structural chunking yields a strictly HIGHER per-section token_iou against a heading's gold_span than a single whole-doc [0..N] block does — the empirical A1 win, not a string. Plus: `python3 tools/crucible.py changeset` (R_NEW_NEEDS_TEST) must pass, locking the new capability to these tests in the crucible.

## Rollback
Single-tool revert: drop `chunk` from the line-184 choices list and the chunk branch in main(); retrieval_eval evaluate path is untouched so no fixture/metric history breaks. Revert library-dev-ingest.md:181-191 to the `DERIVE(raw-text[0..8000], ...)` block (the prior behavior is a pure prefix, no state migration). Delete the new fixture + 3 tests. No data migration, no schema change to existing retrieval_eval fixtures — the chunk output is additive JSON consumed only by the new tests and the ingest program.

## Risk
Low-medium. The parser change is the only edit to a tool other tools import — mitigated by keeping `evaluate` the default and adding `chunk` purely additively (full-suite run in tests catches any import/registry drift). Honesty on scope: this does NOT make library-dev retrieval dense or add real semantic similarity — it makes chunk boundaries structural and metadata real; the A1 'semantic' criterion stays PARTIAL→improved, not fully OK, because true semantic chunking needs embeddings (deferred to a later wave). The biggest real risk is the structural>truncation token_iou assertion being fixture-sensitive; mitigated by hand-authoring chunking_doc.md so a heading's section text is the gold_span and clearly isolated from the doc head that a [0..N] block would capture.

## Proportionality (why this scope)
Right-sized by deliberately folding into retrieval_eval rather than shipping the prompt's nominal standalone tools/chunking.py: tools/ already holds 162 files and the project is actively reducing tool count (honest_callout #3), and retrieval_eval already owns the positional-command dispatch and the RetrievalCase contract a chunk must feed — so +1 subcommand beats +1 file with zero new metric code. Not smaller: a metadata-only stamp without a real boundary splitter would flip the A2 keyword but fail the empirical A1 token_iou gate (the keyword-theatre trap the synthesis names explicitly). Not bigger: no dense index, no rerank, no new persistence layer — those are later waves that each attach to the chunk corpus this lands. Net surface delta: +1 subcommand, +1 fixture, +3 tests, two program edits; one library-dev truncation removed.

# 02 — Leverage plan: RAG retrieval-eval machinery across all of AXON

> Built from a 6-dimension grounded analysis (one analyst per RAG maturity dimension, reading real surface code) + synthesis.
> AXON baseline today: **40/70**. Target path: 40 → ~50 → ~58 → 70.

## The reframe (read this first)

AXON already owns a deterministic, model-free retrieval evaluator (tools/retrieval_eval.py: context_precision/recall/token_iou/faithfulness_proxy/latency, plus a built-but-uninvoked _append_log) and a 70-point audit that flips rows on co-located keywords in named files. The leverage is NOT building dense retrieval first; it is wiring the evaluator's RetrievalCase contract onto the FOUR sparse retrievers AXON already ships (dispatch.tfidf_similarity, agent_memory.recall, library.rank_candidates, shadow search) so every existing surface starts emitting real, gradeable retrieval. Once that single adapter exists, one dense index, one fusion, one rerank/noise-gate, and one CRAG-style verdict each upgrade all six surfaces (library-dev, shadow, dispatch, agent-memory, web_search, code-dev) at once and are each locked by a retrieval_eval metric delta rather than by scaffolding keywords.

**Keystone:** tools/retrieval_trace.py — a live-retrieval → RetrievalCase adapter that captures (query, retrieved[{id,text,score,latency_ms}], answer, relevant_ids, gold_spans) from the existing sparse retrievers and writes the exact JSON shape retrieval_eval._load_case already parses. It needs NO dense substrate, converts the evaluator from a fixtures-only toy into a measurement over live AXON traffic, and is the single dependency under D1/D2/D4, E2/E4/E5/E6/E8, F1/F2/F6, and the eval-locks of every B and C row. B1 (the dense index the prompt nominates) is the Wave-3 keystone, but it is only worth building AFTER this adapter exists to grade it.

## Cross-surface leverage map — how each EXISTING feature gets better + measurable

### library-dev (PDF→cite→report)
- **Improves:** Chunker (Wave 1) replaces the raw-text[0..8000] truncation so citations point at real spans not whole notes; rerank (Wave 2) ranks the correct source first so cite() attaches to the right paper; the F2 verdict blocks an unfaithful citation; dense+HyDE (Wave 3) replaces library.py term-overlap with semantic article ranking.
- **Measured by:** context_precision on rank_candidates output (rerank before/after), faithfulness_proxy per citation (reject gate), token_iou over chunked spans vs gold.

### shadow / code-dev indexing
- **Improves:** Per-chunk sha256 dedup (Wave 1) avoids re-DERIVing whole shadow notes on a one-line edit; code chunks on def/class boundaries (A6) so findings attach to whole functions; freshness gate gains a retrieval-index area; a persistent 'reject' verdict on a stale-shadow-backed query becomes a re-index signal.
- **Measured by:** chunk-level stale_count in freshness check, retrieval-quality drift (E5) tying a degrading shadow note to a re-index trigger, latency_ms on shadow search.

### dispatch routing
- **Improves:** top_matches[:3] (which nobody re-scores today) get reranked → fewer wrong-program dispatches; the bare best_score<threshold fallback becomes an evidence-quality-explained reroute (F2); dense+RRF fusion (Wave 3) fixes the vocabulary-mismatch mis-fires PREFER_COMPILED_FLOOR exists to guard.
- **Measured by:** per-query trace of full candidate scores (E8 why-did-it-route-here), context_precision on top_matches, dispatch-index staleness hours (B4).

### agent-memory recall
- **Improves:** metadata_filters (source/date/confidence) exclude stale/low-confidence memories at recall time; second-pass rerank cuts TF-IDF noise before context load; packing loads MORE relevant memory within budget; dense recall surfaces episodes phrased differently; F6 captures retrieval outcomes so recall earns its keep on retrieval not just program memory.
- **Measured by:** context_precision after the noise gate drops orphan/low-score entries, recall-quality drift as the corpus grows, latency on _tfidf_rank.

### web_search fallback
- **Improves:** The wire that does not exist today gets built: a score_live 'reject' verdict (or all-reject noise gate) triggers a bounded web_search behind the F4 autonomy gate, then re-runs score_live to confirm precision rose; the same B7/F5 confidence gate decides local-vs-web instead of always/never.
- **Measured by:** context_precision/faithfulness_proxy rising above the reject floor after fallback (proves the web call helped), governance-denial test (fallback suppressed under autonomy-off).

### code-dev / workflows (adaptive-free-text.yml)
- **Improves:** The adaptive loop's first step becomes an auditable retrieval plan (F1) instead of an opaque rank; query_planner subqueries feed better candidates; a replayable trace makes retrieval regressions reproducible in CI; should_retrieve gates whether a workflow step retrieves at all.
- **Measured by:** context_recall driven by the plan's expected_relevant_ids, per-subquery provenance trace, trace replay-through-score_case in CI.

## Honest callouts (cost, risk, where I disagree)
- HIGH VALUE, real effect: the keystone retrieval_trace.py adapter, _append_log invocation, rerank, the noise gate, and the F2 verdict→web fallback. These wire numbers onto retrievers users already run; the metric deltas are genuine, not keyword theatre.
- ASPIRATIONAL / keyword-driven risk: roughly a dozen of these rows flip via rag_maturity_audit's _contains_all/_contains_any co-located-string detection (e.g. B3 wants 'reciprocal rank fusion' in a docstring, B6/C1/C2 want 'multi-hop'/'hyde'/'step-back', F-rows want token pairs in one file). It is possible to raise the score WITHOUT raising real quality by planting strings. Defend against this by treating the audit score as necessary-not-sufficient and gating each row on its retrieval_eval METRIC delta (the eval_locks), not the keyword. The crucible/metrics_manifest tripwire discipline (every metric needs a falsifying test) is the right guardrail and must be extended to each new row.
- SURFACE-AREA COST is real and cuts against a stated goal: tools/ already holds 162 Python files and the project is actively trying to REDUCE tool count. This plan proposes ~6-8 new tools (retrieval_trace, chunking, rerank, retrieval_quality, retrieval_index, retrieval_fusion, query_planner). Mitigations: (a) Wave 1/2 maximize edits-to-existing-files over new tools; (b) Wave 3's shared sparse() consolidates THREE divergent TF-IDF copies (dispatch, agent_memory, synapse_suggest) — net surface reduction in that area; (c) several proposed 'new tools' can be subcommands of retrieval_eval/retrieval_index rather than standalone files. Push to fold retrieval_quality+retrieval_trace+drift into retrieval_eval subcommands before accepting them as separate tools.
- SEQUENCING DISAGREEMENT with the prompt's stated keystone: the prompt nominates B1 (dense index) as THE keystone, but every dimension's own keystone_dependency points at the live RetrievalCase adapter, and B1's only honest eval-lock (dump-case fixtures) is itself produced by that adapter. Building B1 first would be a dense index graded only by hand-made fixtures — the exact 'mirage' this codebase's recent commits (multiple-code-dev e2e 'the proof the mirage lacked') warn against. Build the adapter first; B1 is the Wave-3 keystone.
- DENSE INDEX (B1) cost may exceed benefit short-term: AXON's corpus is small (programs + workspace), sparse TF-IDF already scores B2 green, and the sentence-transformers dependency is heavy for a project that prizes model-free determinism. The hashing-vectorizer fallback keeps it deterministic but then 'dense' is barely more than richer sparse. Recommend shipping B1 LAST, behind a measured recall gap from Wave 1 traces that proves sparse is actually losing relevant chunks — if the live context_recall is already ~1.0, defer B1 and bank the score elsewhere.
- DOC/PROGRAM PROLIFERATION: several rows add workspace/programs/*.md neurons (query-expand.md, retrieval-reflect.md) and fixtures. Each is cheap individually but the program registry and dispatch index grow, slightly degrading dispatch precision — a small self-inflicted retrieval cost the freshness/drift gates should watch.

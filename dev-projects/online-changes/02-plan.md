## Thesis

Two outcomes, one plan. LEVERAGE is the rag_maturity_audit score: it sits at a verified 40/70 today (35 rows × 2 pts; `python3 tools/rag_maturity_audit.py audit` returns score:40, max_score:70). REACH is how many surfaces can actually invoke the two deterministic, model-free RAG tools (retrieval-eval, rag-maturity-audit), which today exist only on a staged merge and are callable by exactly nothing beyond a local shell. The unifying argument: you cannot honestly raise the score by capability until you can MEASURE over live traffic, and you cannot measure until the foundation is on main and reachable — so reach and the keystone trace-adapter come first, capability second, and the expensive dense substrate LAST and only if a measured recall gap justifies it. Every row is closed by a real metric/behavior delta in the crucible, never by the audit's grep-able keyword (the audit is necessary-not-sufficient by design). The same move that widens reach over MCP (PR-1) also closes a live remote-write hole, so reach and safety are the same cheap first step. The result: 40→70 on leverage AND shell→MCP→discoverability→in-process/standalone on reach, threaded by a read-only-by-default contract whose single exception (the web arm) is the project's first AEGIS-gated network capability.

## Governance thread

The contract is read-only-by-default with exactly one deliberate, gated exception. PR-1 is the boundary-setting move: it exposes the two tools to every MCP client through the SAFE_TOOLS triad while, in the SAME commit, closing a verified live write hole — today `_is_readonly_call('retrieval-eval',['evaluate','--append-log'])` and `('rag-maturity-audit',['audit','--root','/etc'])` BOTH return readonly:True (verified: _FORBIDDEN_FLAG_TOKENS = ('--out','--output','-o','--write','--workspace','--file','--path','--dest') — neither --append-log nor --root is listed), so a remote client could mkdir+append JSONL under private my-axon/ or retarget the maturity scan. PR-1 appends both tokens and pins READONLY_SUBCOMMANDS to {evaluate}/{audit}, locked by refusal tests that assert subprocess.run never fires. Every Phase-1/2 capability keeps this contract: PR-3's capture is append-only and OFF by default behind read_pref(ws,'retrieval-trace','off'); PR-7's gate/pack are stdin→stdout with no write_state; PR-6's rerank is read-a-case/print-stdout. The ONE place read-only is intentionally broken is PR-8's web path — and that is precisely where AEGIS gating threads in. _policy.md exists today with develop/test-execution/pr-create/merge/build but NO `web:` line (verified); PR-8 adds `web: grant` to _policy.md and registers `'web':'human'` as the fail-closed DEFAULT in aegis_policy.CAPABILITIES, deliberately NOT adding it to GATED={test-execution,merge} (web needs a grant, not a green gate — keeping it orthogonal to the crucible). The retrieval-reflect program calls aegis_policy.resolve(capability='web', policy=load_policy('_policy.md'), grant_active=<grant>) and touches the network ONLY on allowed:True; because the default is 'human' and the grant must be present, autonomy-off provably suppresses the network. This is locked by a governance-DENIAL test (resolve returns allowed:False under grant_active=False AND a monkeypatched web_search sentinel stays uncalled) — the network is the first genuinely gated capability, proven suppressed, not asserted by a keyword. So: PR-1 sets the read-only boundary at the MCP edge; PR-8 opens exactly one path through it, behind AEGIS.

## Sequencing rationale

Reach-and-safety first because it is the cheapest and highest-value: Phase 0 is three S-effort PRs that land the foundation, open it to every MCP client, and close a real remote-write security hole — the widest reach increase per line in the whole plan, and the one phase with a genuine security payload (PR-1's --append-log/--root deny). It claims zero capability points, which is honest: you don't get to bank maturity for merely existing. Measurement strictly precedes capability because the alternative is the 'mirage' the codebase explicitly warns against (commit f3c836f 'the proof the mirage lacked'): PR-3 builds the live-trace adapter so that every later claim is graded against real dispatch/recall traffic, not hand-tuned fixtures. This is why the dense substrate (the nominal keystone) is sequenced LAST and GATED — PR-9 Part B builds the dense index ONLY if Phase-1 live traces show aggregate context_recall < 0.95; on AXON's small corpus where sparse already scores B2 green, the honest likely outcome is DEFER with a recorded measurement. Cheap, model-free precision wins (Phase 2 rerank/gate/pack) come before expensive dense work because they raise the very baseline the dense gain must beat — crediting dense with a lift a reranker already delivered would be dishonest. Throughout, reduce-surface-area is respected by preferring subcommands over new tools: the chunker, rerank, verdict, hybrid, and planner all land as subcommands of retrieval_eval/retrieval_index/library/context (in-SEARCH_GLOBS, reused, not duplicated), and PR-9 Part A is a NET reduction (2 TF-IDF copies → 1). Dense is last not only because it's gated but because it's the densest, riskiest surface — and it depends transitively on PR-3's traces and PR-2's measurement being real first.

# pr-25.5 — `state next` ↔ `_index.md` + pending PRs

**Wave**: W3 · **Goals**: T-S1.12, partial G.wf.07 · **Depends-on**: PR-17 (`_index.md`), PR-9.6 (`next-action` fast path)

## Why (problem statement)
`code-dev next` (post PR-9.6) reads `_meta.next-action` first. But when there is no cached next-action, the fallback reasoning still ignores **two of the most important signals**: stale studies (PR-17 `_index.md`) and in-progress PRs. R5 T-S1.12 closes this loop: `next` should suggest a stale-area study run or resume an in-progress PR when relevant.

## Evidence (from studies)
- `helpers/cd-study-c4-p2-targets.md` → T-S1.12 "`state next` integration: includes ≥ 1 study suggestion when applicable".
- `helpers/cd-study-c2-p4-integration.md` → "state next is the daily-driver verb; should surface study staleness".
- `helpers/cd-wf-c1-p1-canonical-flows.md` → WF3 (resume-flow) starts at `state next`.

## Design notes
- `code-dev-next.md` reasoning (after PR-9.6 fast path miss):
  1. Read `study/_index.md` via `tools/study_index.py.list()`.
  2. For each area with staleness ≥ `stale` and touched by current PR diff (heuristic: filename overlap) → suggest `study --mode=<m> --target=<area>`.
  3. Read `pr_aggregate.list(project)` (PR-9.5) → any PR in `state=in-progress` → suggest resume.
  4. If both → present top 2.
  5. If neither → emit "all caught up" + show 1 idea from `meta board`.
- Tests in `tests/test_next_integration.py`: 3 fixtures (stale study, in-progress PR, both).

## Pitfalls
- Diff-overlap heuristic noisy → tunable threshold; explicit `--no-study-suggest` opt-out.
- Suggestion overload → cap at 2.

## Interface sketch
```text
$ code-dev next
suggested next steps:
  1. resume pr-3 (in-progress) — last phase: 3
  2. run: code-dev study --mode=subsystem --target=src/auth  (study 92d old; pr-3 touches src/auth)
```

# pr-15.5 — Events-bus wiring (`_events.log` ↔ kernel)

**Wave**: W3 · **Goals**: D-A1 (R2 #9, score 5/2 = 2.5) · **Depends-on**: PR-9 (`_session.md` substrate), PR-13 (usage logging schema-adjacent)

## Why (problem statement)
`code-dev` runs in isolation from the kernel event bus. Other AXON subsystems (`igap`, schedulers, hook handlers) cannot observe code-dev events (PR ready, study completed, gate failed). R2 D-A1 ranks this at impact 5, effort 2 — it is the **highest-impact integration item** but pulled behind cheaper-first PRs.

## Evidence (from studies)
- `helpers/cd-c4-p3-improvements.md` Rank 9 → D-A1 "Wire `_events.log` to kernel event bus" I=5/E=2.
- `helpers/cd-c2-p1-internals.md` → "`_events.log` is per-project; kernel's `events.py` has `EMIT/ON` infrastructure but code-dev never publishes".
- `helpers/cd-tools-p1-inventory.md` → `tools/events.py` already exists; topic registry is the integration point.
- `helpers/cd-gap-c1-p3-goals-extracted.md` → ties to G.obs.* observability cluster.

## Design notes
- Define canonical topics:
  - `code-dev.pr.created` / `.ready` / `.merged`
  - `code-dev.study.completed`
  - `code-dev.gate.failed` / `.passed`
  - `code-dev.session.checkpointed` / `.recovered`
- `tools/events.py`:
  - `emit(topic, payload)`: writes one JSONL line to `my-axon/log/events/<YYYY-MM-DD>.jsonl` AND mirrors to per-project `_events.log`.
  - `ON(topic, handler)`: register subscriber (used by `igap` in PR-15.6).
- One synthetic subscriber example: `workspace/programs/code-dev-events-emit.md` declares the topic registry; an example handler logs to stderr when `code-dev.pr.ready` fires.
- All code-dev verbs gain a single `EMIT(...)` call at completion (one line each). Mirror discipline: never double-write — the bus is the single source, the per-project `_events.log` is the projection.

## Pitfalls (from failure-mode catalog)
- Double-logging if both old `_events.log` writes and new bus mirror fire → centralize through single helper.
- Topic name churn → topic registry file (one place to rename).
- Volume: events are short JSONL lines; daily-rolled file scales fine.

## Interface sketch
```text
$ tail -2 my-axon/log/events/2026-05-17.jsonl
{"ts":"2026-05-17T15:01Z","topic":"code-dev.pr.ready","payload":{"project":"axon-master","pr":3}}
{"ts":"2026-05-17T15:01Z","topic":"code-dev.session.checkpointed","payload":{"project":"axon-master","turn":40}}
```

## Spec (canonical)
- **Files**:
  - modified: `tools/events.py`; every code-dev verb gains one EMIT call.
  - new: `workspace/programs/code-dev-events-emit.md` (topic registry + example subscriber).
- **Acceptance**:
  1. Every code-dev verb emits one structured event `{program, project, action, ts}` to the bus.
  2. Kernel `events.py` `ON(<topic>)` handlers can subscribe.
  3. Example handler subscribes to `code-dev.pr.ready` and logs.
  4. No double-writes (one helper).
  5. `tools/lint_paths.py` clean.
- **Rollback**: revert; bus quiet; `_events.log` still written by legacy path.
- **Owner**: AGENT writes; HUMAN runs a `pr ready` once and checks event lands.

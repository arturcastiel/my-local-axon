# pr-31.5 — Recursive program loop detector + tour cross-ref lint

**Wave**: W4 · **Goals**: G.safe.07, G.wf.02 (full) · **Depends-on**: PR-1 (T1)

## Why
F-D3 (recursive program invocation) is detectable at audit time via call-graph analysis; without it, a future EXEC cycle would manifest only as a runtime stack overflow. F-F3 (tour gets stale) gets its second-pass lint here for tighter coverage.

## Evidence
- `helpers/cd-gap-c2-p4-failure-modes.md` → F-D3 (Class D), F-F3 (Class F).
- `helpers/cd-gap-c1-p3-goals-extracted.md` → G.safe.07, G.wf.02.

## Design notes
- `tools/call_graph.py`:
  - Scans every `code-dev*.md` for `EXEC(code-dev-X)` references.
  - Builds DAG.
  - Detect cycles; report cycle path.
- Test `tests/test_programs_md.py` gains `test_no_call_cycles` + `test_call_depth_max`.
- Runtime depth limit: `tools/dispatch.py` enforces max depth 10 → HALT with error if exceeded (safety net even if static check missed).
- Tour cross-ref check: re-asserts PR-1's tour lint with the full umbrella vocabulary in place.

## Pitfalls
- Legitimate "self-EXEC" (rare; treat as cycle and reject) → no legitimate cases identified; reject categorically.

## Interface sketch

```
$ python3 tools/call_graph.py --check
OK: no cycles, max depth = 7 (within 10)
$ pytest tests/test_programs_md.py::test_no_call_cycles
PASSED
```

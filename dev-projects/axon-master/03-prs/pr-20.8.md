# pr-20.8 — Split `code-dev-pr-review` into P1-P9 sub-programs

**Wave**: W3 · **Goals**: T-A2 (R2 #15) — **the structural fix for the RED program** · **Depends-on**: PR-2 (gate so each sub-program is gated), PR-12 (rename-safety)

## Why (problem statement)
`code-dev-pr-review.cmp.md` is the single negative-compression file (PR-2 quarantines it). The root cause is that the source program does **9 distinct phases** in one monolith — compile cannot deduplicate the per-phase preamble + identity lock + EXEC chain across phases. The fix R2 cycle-2 D-PR3/D-C2 and cycle-3 T-A2 all converge on: split into P1..P9 sub-programs, each ≤ 1/3 of the source size, gated GREEN by PR-2. The router becomes a thin dispatcher.

## Evidence (from studies)
- `helpers/cd-c3-p1-tokens.md` line 41 → "split into 3 sub-programs (pr-review-context, pr-review-harmonize, pr-review-execute)". Cycle 4 refined to 9 phases.
- `helpers/cd-c2-p1-internals.md` → "pr-review has 9 phases: P1 summary, P2 diff, P3 risk, P4 style, P5 tests, P6 docs, P7 security, P8 performance, P9 final".
- `helpers/cd-c4-p3-improvements.md` Rank 15, Wave 5 → T-A2 "split pr-review into P1-P9 sub-programs" I=5/E=4 (highest impact, highest effort).
- `helpers/cd-c3-p1-tokens.md` → projection: split brings aggregate compiled bytes ≤ 50% of pre-split (because preamble dedup pays off).

## Design notes
- 9 sub-programs `workspace/programs/code-dev-pr-review-p1.md` ... `p9.md`:
  - **P1 summary** — read PR spec + diff, emit synopsis.
  - **P2 diff** — diff-level changes annotated.
  - **P3 risk** — failure-mode catalog check (PR-7).
  - **P4 style** — code style / project conventions.
  - **P5 tests** — test coverage delta (links to PR-28.5 review-coverage).
  - **P6 docs** — doc impact / cheatsheet update.
  - **P7 security** — secret/CVE scan (PR-5).
  - **P8 performance** — token / runtime impact (PR-13).
  - **P9 final** — synthesize verdict + ready-for-merge decision.
- Router `code-dev-pr-review.md`:
  - Default `--all`: invoke P1..P9 in order.
  - `--phase=N`: invoke single phase.
  - `--from=N`: invoke N..9 (for resume after partial review).
  - Persists progress in `_reviewer-state.json` (PR-20.5).
- Each sub-program independently passes PR-2 GREEN gate (target: ≥ 30% compression per sub-program).
- Rename-safety snapshot updated AFTER the split.

## Pitfalls (from failure-mode catalog)
- Split changes semantics → behavioral T3 fixture (PR-29) covers `pr-review` end-to-end.
- Sub-program drift (P3 grows beyond budget) → PR-20 budget block enforces.
- Cross-phase context shared improperly → `_reviewer-state.json` is the ONLY cross-phase carrier (no implicit context).

## Interface sketch
```text
$ code-dev pr review 3          # default --all
P1 summary ✓
P2 diff    ✓
P3 risk    ✓
…
P9 final   ✓  verdict: ready-for-merge

$ code-dev pr review 3 --phase=4
P4 style only ✓
```

## Spec (canonical)
- **Files**:
  - new: `workspace/programs/code-dev-pr-review-p1.md` ... `p9.md`.
  - modified: `workspace/programs/code-dev-pr-review.md` (becomes a router).
- **Acceptance**:
  1. 9 sub-programs exist with declared single-phase responsibilities.

  4. Aggregate compiled bytes ≤ 50% of pre-split single-file size.
  5. Rename-safety snapshot updated (PR-12).

- **Rollback**: revert; monolithic `pr-review` returns (and PR-2 re-quarantines it).
- **Owner**: AGENT writes; HUMAN runs a real PR review end-to-end and validates each phase.

# pr-32 — Golden study outputs

**Wave**: W4 · **Goals**: G.test.08 · **Depends-on**: PR-8 (study modes), PR-20.7 (evals)

## Why
PR-20.7 evals operate on small fixture repos. PR-32 commits **golden outputs** — expected `study/overview.md`, `subsystems/*.md`, etc. for `tiny-py-cli` and `tiny-ts-lib`. Together these provide ratchet protection: a kernel update that subtly changes study output is caught.

## Evidence
- `helpers/cd-gap-c2-p3-test-surface.md` → "Tier 5: golden outputs".
- `helpers/cd-gap-c1-p3-goals-extracted.md` → G.test.08.

## Design notes
- `tests/fixtures/projects/tiny-py-cli/` and `.../tiny-ts-lib/` — minimal real-ish repos (≤ 30 files each).
- For each fixture × each mode → `expected/study/<mode>.md` committed.
- `tests/test_golden_study.py`: runs study in mock mode, diffs against expected with 80% structural-match threshold.
- Golden regeneration: `python3 tools/study_evals.py --regenerate-goldens` (HUMAN-run, gated on explicit flag).

## Pitfalls
- Goldens drift on intentional improvements → `--regenerate-goldens` documented; PR reviewers compare new vs old golden.
- Test flakiness → use mock-model with deterministic fixture responses.

## Spec
- **Files**: new `tests/fixtures/projects/{tiny-py-cli, tiny-ts-lib}/`, `tests/test_golden_study.py`.
- **Acceptance**:
  1. 2 fixture repos, 3 modes each → 6 golden study files.
  2. Test passes at 80% structural-match.
  3. Regeneration script exists.
  4. `tools/lint_paths.py` clean.
- **Rollback**: revert.
- **Owner**: AGENT writes; HUMAN reviews initial goldens.

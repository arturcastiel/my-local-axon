# pr-14 — Router stubs (full 10-umbrella set)

**Wave**: W2 · **Goals**: G.umb.01 (full R3 set) · **Depends-on**: PR-12 (rename-safety harness)

## Why (problem statement)
R3 proposes 10 umbrella verbs that collapse the user-visible surface from 57 to 10 (`lifecycle / state / journal / pr / review / shape / safety / knowledge / flow / meta`). Today only `code-dev review` is a partial umbrella; the rest are flat. R3 mandates that routers ship **first**, additively, before any rename — old verbs continue working as aliases. This PR ships all 10 routers (the originals plus the 5 not in earlier plan drafts: `review / journal / knowledge / flow / shape`).

- `helpers/cd-tools-p3-migration.md` → migration wave plan: routers Wave-1 → aliases Wave-2 → renames Wave-3+.
- `helpers/cd-wf-c4-p2-roadmap.md` Wave 1 PR-1.1..1.10 → "10 PRs, one per umbrella (independent)".

- `helpers/cd-gap-c1-p3-goals-extracted.md` → G.umb.01 (10 verb routers, additive).

## Design notes
- 10 router files (kept tiny, ≤ 1 KB each):
  - `code-dev-lifecycle.md` — new / init / load / tour
  - `code-dev-state.md` — status / next / resume / handoff / metrics / save / restore / undo / actions
  - `code-dev-journal.md` — log / decision / event / search
  - `code-dev-pr.md` — create / update-spec / link / respond / review / ready / github / import / list / archive / stack
  - `code-dev-review.md` — already partial; gains `--mode={diff,scope,self,tests,coverage}`
  - `code-dev-shape.md` — combine / divide / partition / phase / plan-master / link
  - `code-dev-safety.md` — freeze / thaw / dont-do / preflight / audit
  - `code-dev-knowledge.md` — study / shadow / explain / impact / reviewer-track
  - `code-dev-flow.md` — plan / merge / cascade / changelog / test-map / finalize
  - `code-dev-meta.md` — whatif / help / actions / context / cheatsheet / dry-run / examples
- Each router dispatches by sub-command using existing programs unchanged. Example:
  ```
  # PROGRAM: code-dev-pr
  # desc:    PR lifecycle umbrella — list/create/update-spec/link/respond/review/ready/github
  CASE($1):
    create       → EXEC(code-dev-pr-create $@)
    update-spec  → EXEC(code-dev-pr-update-spec $@)
    list         → EXEC(code-dev-pr-list $@)
    review       → EXEC(code-dev-pr-review $@)
    …
  ```
- Existing top-level verbs (`code-dev review`, `code-dev pr 3`, etc.) keep working: `code-dev.md` keeps its current dispatch table but **also** routes through the umbrella as a second path. Both paths are recorded in `_actions.log` so usage drives PR-26..28 rename decisions.
- Old → new aliases ARE NOT introduced this PR; that's PR-26..28 (renames) territory. This PR is **purely additive**.
- Rename-safety snapshot (PR-12) re-taken AFTER routers land so future PRs reference current state.

## Pitfalls (from failure-mode catalog)
- **F-D1 free-text routed to wrong verb** → out of scope; PR-18/19 measures.
- **F-D2 stub forwards to wrong target** → snapshot test (PR-12) catches.
- **F-D3 recursion** → out of scope; PR-31.5 detects.
- Old verbs break → invariant: this PR is additive, never removes a verb.

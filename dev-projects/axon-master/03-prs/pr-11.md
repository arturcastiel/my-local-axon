# pr-11 — Plan reads rules (full) + governance trace + `--rule` injection

**Wave**: W2 · **Goals**: G.plan.04, G.plan.05, T-S1.11 · **Depends-on**: PR-4

## Why (problem statement)
PR-4 emitted an empty trace. This PR makes `code-dev plan` actually consult `safety/rules.md` during option generation: each candidate PR is checked against rules and either filtered or annotated. R5 plan-mode `constrained` also requires `--rule "..."` ad-hoc injection so users can express a one-shot constraint without permanently editing `rules.md`.

## Evidence (from studies)
- `helpers/cd-study-c3-p1-plan-modes.md` → mode `constrained` design: `--rule "<text>"` echoed in trace; treated as a temporary R-INLINE-* rule.
- `helpers/cd-study-c4-p2-targets.md` → T-S1.11 acceptance: "each --rule echoed in plan output 'Rules honored' section; PR proposals filtered".
- `helpers/cd-gap-c1-p3-goals-extracted.md` → G.plan.04 (plan reads rules), G.plan.05 (governance trace populated).
- `helpers/cd-gap-c3-p1-governance.md` → "plan is the natural place to surface rules — output is human-reviewed".

## Design notes
- `code-dev-plan.md` modified:
  - Before option ranking, calls `tools/rules.py.applicable(active_rules + inline_rules)`.
  - Each rule's `applies-to` field (PR-4 schema) decides whether the rule fires on this plan.
  - Each PR candidate annotated `{filtered: bool, reason: <rule-id>}`.
  - Filtered count → trace; if > 80% filtered, HALT and QUERY user (mass-rejection means the user probably set a rule wrong).
- `--rule "<text>"`:
  - Parsed into an in-memory `R-INLINE-<n>` rule with `scope: project, applies-to: plan`.
  - Echoed under `## Governance trace · Ad-hoc rules` subsection.
  - Not persisted (single-run only).
  - Multiple `--rule` flags allowed.
- Synthetic test: rule "no PRs that touch `tools/compile-write.py`" filters PR-2 from a plan that would have included it.

## Pitfalls (from failure-mode catalog)
- **F-E1 plan violates rules** → primary fix.
- **F-E3 rule contradiction** → if two inline rules conflict, HALT (delegates resolution to user).
- Mass-rejection (>80% filtered) → HALT (catches malformed rules).

## Interface sketch
```text
$ code-dev plan --rule "no PRs that touch tools/compile-write.py"
…
## Governance trace
loaded: 0 rules from safety/rules.md
ad-hoc rules:
  R-INLINE-1: no PRs that touch tools/compile-write.py
filtered: 1 (pr-2 — R-INLINE-1)
flagged:  0
conflicts: 0
```

## Spec (canonical)
- **Files**:
  - modified: `workspace/programs/code-dev-plan.md`, `tools/rules.py`.
- **Acceptance**:
  1. Synthetic rule filters at least one PR candidate.
  2. Trace shows filtered count + reasons.
  3. HALT triggered if > 80% of candidates filtered.
  4. `--rule "..."` parses; echoed in trace as `R-INLINE-<n>`.
  5. Multiple `--rule` flags accumulate.
  6. `tools/lint_paths.py` clean.
- **Rollback**: revert; plan ignores rules again (governance trace still emits, empty).
- **Owner**: AGENT writes; HUMAN exercises with a real rule.

## Codebase grounding
- **modify**: [`workspace/programs/code-dev-plan.md`](../../../../workspace/programs/code-dev-plan.md) — in `## LOAD CONTEXT` block, after reading `_meta`, also `rules ← TOOL(rules, load)`; apply against generated option set in the codebase-analysis block (~line 80+).

- **HALT**: when `filtered / total > 0.80`.

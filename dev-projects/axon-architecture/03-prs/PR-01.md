# PR-01 · MCP read-only invocation schema (Theme D · security) · AUTO

## Closes
- **F13** — `dispatch-stats precision --out <path>` = arbitrary file write through a "read-only" allowlist.
- **F14** — `usage prune` = destructive log rewrite; `usage record` = forged telemetry — same allowlist.

## Root cause
`mcp_server.handle_call` (tools/mcp_server.py:103-132) forwards `arguments.args` verbatim as argv with
**no subcommand or flag filtering**. The `SAFE_TOOLS` allowlist gates *which tool* runs, never *what it
does*. So any exposed tool's mutating subcommand or path-bearing flag is reachable by an external MCP
client.

## Change (tools/mcp_server.py)
1. `READONLY_SUBCOMMANDS = {"dispatch-stats": {"summary","savings"}, "usage": {"top","suggest","aggregate"}}`
   — for a tool listed here, the call's first positional (the subcommand) MUST be in the set.
2. `_FORBIDDEN_FLAG_TOKENS = ("--out","--output","-o","--write","--workspace","--file","--path","--dest")`
   — rejected on **any** exposed call (blocks write-redirect + state-retarget for all 13 exposed tools).
3. `_is_readonly_call(name, cli_args) -> (ok, reason)`; `handle_call` returns `isError:True` + reason
   before running the subprocess when not ok.

## Scope decision
The global forbidden-flag guard covers the write-via-flag vector for **all** exposed tools; the per-tool
subcommand allowlist covers the write-via-subcommand vector for the **two named offenders**. Extending an
explicit read-only subcommand allowlist to the other 11 exposed tools is a follow-up hardening (no other
exposed tool has a *named* write subcommand in the study); the guard fails safe meanwhile.

## Tests (tests/test_mcp_server.py)
- `dispatch-stats precision --out /tmp/x` → refused (subcommand + flag); file NOT written.
- `usage prune --before 1d` → refused; `usage record …` → refused.
- `dispatch-stats summary` / `usage top` → allowed (pass the gate).
- a forbidden flag on any tool (e.g. `health --out x`) → refused.
- `--workspace` anywhere → refused.

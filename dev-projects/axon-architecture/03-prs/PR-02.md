## CI-topology reality (important)
The remote is GitLab (`ci.tno.nl`) and there is **no `.gitlab-ci.yml`**; prior merges reported
"No pipeline running". So `.github/workflows/ci.yml` is **inert on this remote** (it would bite only a
GitHub mirror). True mechanical execution on this remote requires an **owner decision**: a runner-backed
`.gitlab-ci.yml`, or a local pre-push hook (`pre-commit install --hook-type pre-push`). This PR therefore
makes the *intent* mechanical and lock-tested, and documents the runner decision rather than risk
blocking merges by adding a pipeline with no runner.

## Change
1. `.github/workflows/ci.yml` — add a `crucible-gate` job running `python3 tools/crucible.py gate` (deps
   installed like `tests-full`). Authoritative gate wherever GitHub Actions runs.
2. **`tests/test_ci_runs_crucible.py`** — lock-test (mechanical NOW, since pytest runs inside crucible +
   locally): assert `.github/workflows/ci.yml` declares `crucible.py gate`; assert every BLOCK control in
   `crucible.json` has a runnable `cmd`. A regression (removing the job, or a BLOCK control with no cmd)
   goes red.

## Why the conformance test is the real teeth here
Because CI may be inert on the live remote, the lock-test — run by the crucible gate itself and by every
local `pytest` — is what actually prevents the keystone from silently regressing. It encodes "CI must run
crucible" as a self-checking invariant independent of which CI executes.

## Tests
- `test_ci_declares_crucible_gate` — ci.yml contains `crucible.py gate`.
- `test_every_block_control_has_cmd` — each crucible.json control with severity BLOCK has a non-empty cmd.

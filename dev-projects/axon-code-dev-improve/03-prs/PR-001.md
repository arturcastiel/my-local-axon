# PR-001 — Collapse confirmed duplicate DAG emitter + cascade
> Phase: pr · Status: spec-ready · Maps to: prerequisite refactor (Step 1) · Depends on: PR-000 (re-verify, done)
> Risk: low — no user-facing behaviour change · Gate: tests green (Core Rule 13)

## Objective
Remove two confirmed structural liabilities that would corrupt R4/R5 if left in place, by collapsing duplicate graph machinery onto the single `dag.py` engine. Invisible to the owner; it is the safe foundation the reconciler (PR-004) stands on.

## Source-grounded scope (verified 2026-06-24)

### Item A — `plan_dag.py` must emit the canonical `axon-dag` schema via `dag.py` primitives  ✅ real
**Problem (confirmed):** `tools/plan_dag.py:run()` builds its own `result` dict — `{"ok", "nodes":[id-strings], "edges":[[from,to]], "topo", "critical_path", "wave_count"}` (no `schema`, no `schema-version`, nodes as id-strings) — and writes it to `03-prs/DAG.json` via `from _axon_io import atomic_write` (a second write path that bypasses `dag.py`). This is structurally incompatible with `dag.py:verify()` (requires `schema:"axon-dag"`, `schema-version`, nodes as dicts `{id,kind,name,label,status,child-dag}`, edges as `{from,to,kind}`). Two emitters → one filename → two shapes.

**Change:** Rewrite `plan_dag.run()` to build the DAG with `dag.make_empty(level="plan", owner=…)` → `dag.add_node(d, node_id=…, kind="pr", name=…, status=…)` per PR → `dag.add_edge(d, frm=dep, to=id, kind="depends")` per edge → persist with `dag._save_and_render(prs_dir/"DAG.json", d)` (which also regenerates `DAG.md`). Drop the bespoke `result`-dict write and the `atomic_write(DAG.json …)` call. Keep `topo`/`critical_path`/`wave_count` only as the function's *return value* (for callers/CLI), never as the on-disk shape.
- Files: `tools/plan_dag.py` (run + render path); remove now-dead `render_md`/`emit_mermaid` if `dag._save_and_render` supersedes them (confirm no other caller first).
- Preserve: the case-insensitive `PR-*.md` glob fix; the cycle-detection early return (map to `dag.verify` cycle reporting).

### Item B — `phase_model.stale_downstream` and `dag.cascade_stale` share one cascade core  ✅ real (with nuance)
**Problem (confirmed):** `tools/phase_model.py:335 stale_downstream()` reimplements the exact backward cascade-invalidation of `tools/dag.py:219 cascade_stale()` (mark every non-`pending` transitive dependent `stale`). Divergence risk under partial repair.
**Nuance (verified):** they operate on **different structures** — `stale_downstream` over the `_phases.json` manifest (`phases[]` with `deps`), `cascade_stale` over a DAG dict (`nodes`/`edges`). A naive delegate is wrong.
**Change:** Extract the algorithm into ONE pure helper, e.g. `dag.cascade_stale_ids(descendants_fn, status_of, set_stale) -> list[str]` (or a small `_graph.py` core: `descendants(adj, id)` + `cascade_stale(adj, status_get, status_set)`). Have BOTH `dag.cascade_stale` and `phase_model.stale_downstream` call it, each passing its own adjacency/status accessors. Behaviour identical; one implementation.
- Files: `tools/dag.py`, `tools/phase_model.py` (+ optional `tools/_graph.py`).

### Item C — `workflow_dag.py` consolidation  ⏸ DEFERRED (verification correction)
**Re-audit result:** `tools/workflow_dag.py` is **live** (`REGISTRY.json:253`), **tested** (`tests/test_workflow_dag.py`, also exercised by `test_phase5_pilot`, `test_tool_invocation_smoke`, `test_workflow_suite`) — **NOT a dangling pointer.** The council's "reconcile registry/test" sub-item and the "crucible pytest collection fail-open" risk are **moot**.

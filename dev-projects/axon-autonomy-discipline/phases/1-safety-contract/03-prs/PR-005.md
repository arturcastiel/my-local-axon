# PR-005 — Cadence mechanism (operate-loop, part 1)

- **Status:** spec
- **Phase:** 1-safety-contract
- **Complexity:** M  (reduced from L — see Scope note)
- **Depends on:** PR-002 (the reanchor), PR-004 (the breaker pattern)
- **Why:** The reanchor cadence ("re-anchor every 5 commands") needs a state machine to know *when* it is
  due and *whether it lapsed*. This builds that state; PR-006's backstop turns it into enforcement.

## Scope note (single-concern, minimal core-touch — the discipline applied to itself)
The plan's PR-005 bundled a `dispatch.py` command counter + an orchestrator auto-fire + the
`autonomy-discipline.yml` workflow — four concerns across two CORE files. Refined:
- **No `dispatch.py` change.** The cadence keys off `W:turn-count` — already incremented per turn by the
  kernel/OUTPUT-LAYER (the enforced counter the deep study found). In an autonomous run each turn ≈ a
  command, so "5 turns since the last reanchor" is a faithful, zero-core-touch proxy for "5 commands."
- **No orchestrator change.** The ENFORCEMENT is PR-006's backstop (a control that BLOCKS when the
  reanchor lapsed). An orchestrator auto-fire would only be a *convenience* (help the agent remember); the
  backstop makes the cadence enforced-not-trusted without it. Deferred (optional).
- **The `autonomy-discipline.yml` workflow** (compose contract→reanchor→select→breaker) is the
  self-hosting composition — separable from enforcement. Deferred to a thin follow-up.

So PR-005 = the cadence state machine + the reanchor recording its own fire. PR-006 = the backstop (the
teeth). This is the mechanism/signal split again, kept single-concern.

## Acceptance criteria
1. `autonomy_cadence.record_reanchor(ws)` stores the current `W:turn-count` as `last_reanchor_turn`.
2. `autonomy_cadence.since_reanchor(ws)` = `turn-count − last_reanchor_turn` (turn-count read from W:).
3. `autonomy_cadence.should_reanchor(ws, every=5)` → True when `since_reanchor ≥ every`.
4. The reanchor program (`autonomy-reanchor.md`) records its fire — `TOOL(autonomy-cadence, record-reanchor)`
   — so the cadence state tracks the last reanchor. Asserted statically + the tool effect tested.
5. No reanchor ever recorded → `since_reanchor` = turn-count (treated as "due") — so a never-reanchored
   autonomous run is flagged by the backstop (PR-006), not silently fresh.

# pr-19 — Dispatch quality metric

**Wave**: W3 · **Goals**: G.obs.06 · **Depends-on**: PR-18 (corpus), PR-13 (usage)

## Why (problem statement)
PR-18 produces the data; PR-19 surfaces it. Without `code-dev meta dispatch-stats` the corpus output is dead. The Wave-3 entry/exit gates and the Wave-4 final gate both reference "P@1 ≥ 0.8" — this is the program that reports it.

## Evidence (from studies)
- `helpers/cd-gap-c1-p3-goals-extracted.md` → G.obs.06.
- `helpers/cd-gap-c4-p2-priority-matrix.md` → dispatch quality measurement is the #1 unmeasured observability lever.

## Design notes
- `tools/dispatch_stats.py` gains:
  - `summary(since=<date>)`: returns `{P@1, P@3, recent_failures: [...]}`.
  - `record(test_run, P@1, P@3)`: appends to daily JSONL.
- `workspace/programs/code-dev-meta-dispatch-stats.md`: prints summary + 5 most-recent failures with prompt + actual vs expected.
- Targets (advisory, not enforced):
  - P@1 ≥ 0.8 (warn below)
  - P@3 ≥ 0.95
- Failures table feeds the next iteration of the corpus (add adversarial phrasings).

## Pitfalls (from failure-mode catalog)
- Targets too tight at first run → advisory only (warn, not block); becomes Wave-4 gate.

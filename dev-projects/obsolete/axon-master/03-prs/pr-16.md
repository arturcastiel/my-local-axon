# pr-16 — Plan modes (4) + `--budget N`

**Wave**: W2 · **Goals**: G.plan.01, G.plan.02 (full), T-S1.10 · **Depends-on**: PR-4 (governance scaffolding), PR-11 (rule reading)

## Why (problem statement)
`code-dev plan` today has a single output shape (execution-oriented). R5 plan-modes study identified that users actually want **at least 4 distinct planning lenses**: tactical (default), strategic (multi-wave), operational (cost/timing), decision (multi-option). Plus, plans need a **token/PR budget** (`--budget N`) so they stop ballooning into 50-PR mega-plans (this very project hit that on plan v4).

## Design notes
- 4 modes shipped:
  - `tactical` (default): today's behavior; per-wave PR breakdown with acceptance.
  - `strategic`: high-level wave map only, no per-PR detail; ≤ 800 tokens.
  - `operational`: focuses on time + token estimates; ranks by `est-tokens, est-hours`.
  - `decision`: multi-option layout — 2-3 candidates per problem with trade-offs.
- `--budget N`:
  - Cap output at N PRs.
  - Overflow → `02-prs.deferred.md` (sibling file, same project).
  - Honored by all modes.
- `--mode=...` echoed in plan header; `_meta` records last-mode for `replan` later (post-1.0).
- `code-dev-plan.md` becomes a mode dispatcher; per-mode rendering remains inline (modes are similar enough to share most code paths).

## Pitfalls (from failure-mode catalog)
- **F-C3 token-budget overflow** → `--budget N` is the explicit user-facing knob.
- Plan v1 wrong post-execution → out of scope; replan path defined in 03-plan.md execution semantics.
- Mode confusion → default `tactical` matches existing behavior (no surprise on omission).

## Interface sketch
```text
$ code-dev plan --mode=strategic
# Strategic plan — axon-master
Wave 1 (foundation, 7 PRs)
Wave 2 (modes + sessions, 13 PRs)
Wave 3 (observability + perf, 15 PRs)
Wave 4 (renames + behavioral, 13 PRs)

$ code-dev plan --budget 10
✓ wrote 02-prs.md (10 PRs)
✓ wrote 02-prs.deferred.md (38 PRs)
```

# U-7 — Plan / study blanket budget vs per-mode reconciliation

**Wave**: U.C · **Severity addressed**: S2 (F-011, F-019) · **Depends-on**: —
**Owner**: AGENT · **Status**: ready

## Why

`code-dev-plan.md` and `code-dev-study.md` each declare a blanket
`# budget: output-cap: N` and *also* a `# modes:` (or `# plan-modes:`) block
with per-mode caps that exceed the blanket. E.g. plan tactical mode declares
`output-cap: 6000` while the blanket is `2000`. `budget_lint` accepts both
silently. Documentation drift.

Two acceptable resolutions:
1. Drop the blanket `output-cap` in favor of the per-mode block (when
   present).
2. Add an explicit `# modes: (overrides blanket # budget: caps above)`
   comment to document the override.

We choose option 2 — it's strictly additive (one comment line) and doesn't
change runtime behavior or risk breaking any caller that reads the blanket.

## Design notes

- Single comment line above each `# modes:` / `# plan-modes:` block.
- No code change. `budget_lint` continues to validate both blocks as it does today.
- A future PR could extend `budget_lint` to warn on per-mode > blanket; out of
  scope for this errata.

## Pitfalls

- **F (doc-only fix)**: comment lines drift if the modes block is reordered.
  Low risk; modes block is at top of file.

## Interface sketch

```
# budget:
#   input-cap:    8000
#   output-cap:   2000
#   cache-prefix: 2048

# modes: (overrides blanket # budget: caps above when --mode is set)
#   overview:    {input-cap:  8000, output-cap:  4000}
#   ...
```

## Spec

### Files

- `workspace/programs/code-dev-plan.md` — add comment above `# plan-modes:`.
- `workspace/programs/code-dev-study.md` — add comment above `# modes:`.

### Diff (plan)

```diff
+# plan-modes: (overrides blanket # budget: caps above when --mode is set)
-# plan-modes:
 #   tactical:    {input-cap: 4000, output-cap: 6000}
 ...
```

## Codebase grounding

- The `# modes:` and `# plan-modes:` block syntax was introduced in PR-30
  with the parser at `tools/budget_lint.py`. The parser is regex-based and
  ignores arbitrary comments; the override comment will not affect linting.

# U-13 — `code-dev plan --mode=decision` writes `03-decisions/adr-NNN-<slug>.md`

**Wave**: U.E · **Depends-on**: U-1 (only — sidecar to the main hierarchy) · **Status**: ready

## Why

Decision mode today only prints "2-3 candidate plans with trade-offs" to
stdout. The decision evaporates after the chat ends. ADRs (architectural
decision records) are the standard pattern; promote decision mode to write
a durable `03-decisions/adr-NNN-<slug>.md`.

## Evidence

- `workspace/programs/code-dev-plan.md` MODE DISPATCH `IF plan-mode ≡
  "decision"` branch — stdout only.
- Industry-standard ADR template (Michael Nygard) is the basis for ours.

## Design notes

- New template `workspace/templates/v4-adr.md` with sections:
  1. Title (verb-noun phrase)
  2. Status (proposed / accepted / superseded-by-NNN / withdrawn)
  3. Context — what forces are at play
  4. Decision — what we chose
  5. Alternatives considered (with reasons rejected)
  6. Consequences (positive + negative + neutral)
  7. Related PRs / phases
- Numbering: `adr-001`, `adr-002`, etc. — global per project.
- ADRs are a sidecar: they may exist without phases or PRs, and vice versa.
- `code-dev plan --mode=decision` is interactive — asks for title, context,
  candidates, then writes the ADR.

## Pitfalls

- **R (ADR sprawl)**: decisions get logged for trivial choices. Mitigation:
  template prompt explicitly asks "is this durable and load-bearing?"; if
  no, suggest a journal-decision entry instead.
- **F (renumbering)**: never renumber. Superseded ADRs stay numbered; the
  `Status:` line points to the superseding ADR.

## Interface sketch

```bash
$ code-dev plan --mode=decision
Title: switch to ESLint-flat-config
Context: ...
Candidate 1: keep legacy .eslintrc
Candidate 2: migrate to eslint.config.js
Candidate 3: vendor-lock ESLint version
Decision: candidate 2
Consequences: +modern; -learning-curve

✓ wrote 03-decisions/adr-003-eslint-flat-config.md
```
